# 第七部分：面向未来的演进治理

治理层是架构仓库的"宪法与法律体系"——它定义了架构决策如何被制定、记录、执行和演进。没有治理层，四个视角的内容会随时间漂移、腐化，最终失去可信度。

---

## 1. 治理层 README

```markdown
# 06-governance - 架构治理层

## 使命

确保架构决策的**透明性、一致性、可追溯性**，让架构随业务演进而非随意漂移。

## 核心组件

| 组件 | 用途 | 更新频率 |
|------|------|----------|
| **架构原则** | 指导所有架构决策的基本信条 | 低频（季度评审） |
| **ADR（架构决策记录）** | 记录每个重要架构决策的上下文、决定和后果 | 每次重大决策 |
| **变更管理流程** | 定义架构变更的提案→评审→批准→实施流程 | 低频 |
| **评审检查清单** | 设计评审和代码评审的架构维度检查项 | 中频 |
| **适应度函数** | 可量化、可自动化的架构守护指标 | 持续运行 |
| **技术雷达** | 技术选型的评估与生命周期管理 | 季度更新 |

## 治理理念

```
          轻量级 ←————————————————→ 重量级
              ↑                        ↑
          适合创业期                 适合合规行业
              
     我们的定位：偏轻量，但关键决策必须留痕
```

**核心信条：**
- 治理是赋能而非管控
- 记录决策比强制执行更重要
- 自动化检查优于人工审批
- 原则指导而非规则束缚
```

## 2. 架构原则

```yaml
# 06-governance/principles/architecture-principles.yaml

architecture_principles:
  version: "2.0"
  last_reviewed: "2024-01-15"
  next_review: "2024-04-15"
  approved_by: "Architecture Board"

  # ====================================================================
  # 元原则：关于原则本身的原则
  # ====================================================================
  meta_principles:
    - id: "MP-01"
      name: "原则数量克制"
      statement: "核心原则不超过 12 条，每条必须可操作"
      rationale: "过多原则等于没有原则"

    - id: "MP-02"
      name: "原则必须可验证"
      statement: "每条原则至少有一种方式可以验证是否被遵守"
      rationale: "不可验证的原则只是口号"

  # ====================================================================
  # 核心架构原则
  # ====================================================================
  principles:

    # ------ 业务对齐 ------
    - id: "AP-01"
      name: "业务能力驱动"
      category: "business-alignment"
      statement: >
        系统边界应与业务能力边界对齐，而非按技术层次划分。
      rationale: >
        业务能力相对稳定，按此划分可减少跨团队协调成本，
        提高业务变更的局部性。
      implications:
        - "微服务边界由限界上下文决定"
        - "团队按业务领域而非技术栈组织"
        - "数据库按服务隔离，不共享"
      verification:
        automated: true
        method: "检查每个微服务是否映射到唯一限界上下文（CR-001/CR-002）"
      exceptions:
        - "纯技术基础设施服务（如 API Gateway）不受此约束"
      related_adrs: ["ADR-001", "ADR-005"]

    - id: "AP-02"
      name: "演进式架构"
      category: "business-alignment"
      statement: >
        架构应支持增量变更，避免大爆炸式重构。
      rationale: >
        业务需求持续变化，架构必须能够安全地逐步演进。
      implications:
        - "优先选择可逆决策"
        - "使用 Feature Flag 控制新功能发布"
        - "保持向后兼容的 API 版本策略"
        - "拆分和合并服务都应有渐进方案"
      verification:
        automated: false
        method: "架构评审时检查变更方案是否包含渐进路径和回滚计划"
      related_adrs: ["ADR-003"]

    # ------ 数据治理 ------
    - id: "AP-03"
      name: "数据所有权明确"
      category: "data-governance"
      statement: >
        每个数据实体有且仅有一个权威来源服务。
      rationale: >
        多源写入导致数据不一致，是分布式系统最常见的问题。
      implications:
        - "其他服务通过 API 或事件获取数据副本"
        - "副本数据标记为只读，允许最终一致"
        - "数据实体的 Schema 变更由所有者服务负责"
      verification:
        automated: true
        method: "一致性检查规则 CR-003"
      related_adrs: ["ADR-007"]

    - id: "AP-04"
      name: "事件作为一等公民"
      category: "data-governance"
      statement: >
        跨服务的状态传播优先使用领域事件，而非同步调用或共享数据库。
      rationale: >
        事件驱动降低服务间耦合，提高系统弹性和可扩展性。
      implications:
        - "领域事件需要有明确的 Schema 和版本"
        - "事件存储需要保证至少一次投递"
        - "消费者需要实现幂等处理"
      verification:
        automated: true
        method: "检查跨服务调用中同步 API 与异步事件的比例"
        threshold: "异步事件占跨服务通信 ≥ 60%"
      related_adrs: ["ADR-004"]

    # ------ 技术卓越 ------
    - id: "AP-05"
      name: "API 优先"
      category: "technical-excellence"
      statement: >
        所有服务能力必须通过定义良好的 API 暴露，API 设计先于实现。
      rationale: >
        API 是服务间的契约，先定义契约可以并行开发、提前集成测试。
      implications:
        - "使用 OpenAPI/AsyncAPI 规范定义 API"
        - "API 变更需要版本管理和兼容性评估"
        - "内部服务间也需要正式 API 定义"
      verification:
        automated: true
        method: "CI 检查所有服务是否有对应的 API 规范文件"

    - id: "AP-06"
      name: "故障隔离"
      category: "technical-excellence"
      statement: >
        单个组件的故障不应导致整个系统不可用。
      rationale: >
        分布式系统中故障是常态，必须设计容错机制。
      implications:
        - "服务间调用必须有超时、重试、熔断"
        - "关键路径需要有降级方案"
        - "避免级联依赖超过 3 层"
      verification:
        automated: true
        method: "适应度函数 FF-003（故障隔离度）"

    - id: "AP-07"
      name: "可观测性内建"
      category: "technical-excellence"
      statement: >
        每个服务必须内建日志、指标、链路追踪三大可观测性支柱。
      rationale: >
        分布式系统的调试依赖可观测性，事后补装成本极高。
      implications:
        - "统一日志格式（JSON 结构化）"
        - "标准指标暴露（Prometheus 格式）"
        - "分布式链路追踪（OpenTelemetry）"
        - "每个 API 端点有 SLI 指标"
      verification:
        automated: true
        method: "部署检查清单自动验证"

    # ------ 安全 ------
    - id: "AP-08"
      name: "零信任安全"
      category: "security"
      statement: >
        服务间通信默认不信任，必须经过认证和授权。
      rationale: >
        内网不等于安全网，横向移动是常见攻击路径。
      implications:
        - "服务间使用 mTLS"
        - "所有 API 调用携带身份令牌"
        - "最小权限原则"
      verification:
        automated: true
        method: "网络策略扫描 + API Gateway 配置检查"

    # ------ 团队自治 ------
    - id: "AP-09"
      name: "团队自治与对齐"
      category: "organization"
      statement: >
        团队在服务边界内拥有技术选型自主权，但必须遵守跨团队接口标准。
      rationale: >
        自治提高效率和创新，标准化接口确保互操作性。
      implications:
        - "团队可自选编程语言、框架、数据库"
        - "但 API 格式、事件格式、监控标准必须统一"
        - "共享库只限于接口定义和工具类"
      verification:
        automated: false
        method: "季度架构评审检查"

    - id: "AP-10"
      name: "文档即代码"
      category: "organization"
      statement: >
        架构文档与代码同仓管理，使用版本控制，通过 CI 验证一致性。
      rationale: >
        分离的文档必然过时，与代码同步管理才能保持活力。
      implications:
        - "架构仓库使用 Git 管理"
        - "文档变更走 PR 评审流程"
        - "自动化检查文档与实际系统的一致性"
      verification:
        automated: true
        method: "CI 流水线中的一致性检查"
      related_adrs: ["ADR-002"]
```

## 4. ADR 示例

```yaml
# 06-governance/adrs/active/ADR-004-event-driven-cross-service-communication.yaml

adr:
  id: "ADR-004"
  title: "采用事件驱动架构进行跨服务通信"
  status: "accepted"
  date_proposed: "2024-01-05"
  date_decided: "2024-01-12"
  deciders:
    - name: "张三"
      role: "首席架构师"
    - name: "李四"
      role: "平台团队负责人"
    - name: "王五"
      role: "订单域技术负责人"
  owner: "张三"

  tags:
    scope: "system-wide"
    category: "pattern"
    impact_level: "high"
    reversibility: "costly-reversible"
    affected_views:
      - "business"
      - "technical"
      - "data"

  context: |
    随着微服务数量增长到 15+，服务间同步 REST 调用带来了以下问题：
    
    1. **级联故障**：订单服务调用库存服务超时，导致支付服务也超时
    2. **紧耦合**：添加新的下游消费者需要修改上游服务代码
    3. **性能瓶颈**：同步调用链路过长，P99 延迟超过 3 秒
    4. **数据一致性**：分布式事务（2PC）实现复杂且性能差
    
    当前系统中约 80% 的跨服务通信是同步 REST 调用。

  decision: |
    我们决定采用**事件驱动架构（EDA）**作为跨服务通信的主要模式：
    
    1. **消息中间件**：选用 Apache Kafka 作为事件总线
    2. **事件格式**：使用 CloudEvents 规范 + Avro 序列化
    3. **Schema 管理**：使用 Confluent Schema Registry
    4. **适用范围**：
       - 跨限界上下文的状态传播 → 必须使用事件
       - 查询类操作 → 允许同步 REST/gRPC
       - 同一限界上下文内部 → 团队自行决定
    5. **一致性模型**：最终一致性 + Saga 模式处理分布式事务

  alternatives:
    - name: "维持现状（同步 REST）"
      description: "继续使用 REST 调用，增加熔断和重试"
      pros:
        - "无需引入新技术栈"
        - "调试和追踪相对简单"
      cons:
        - "无法根本解决级联故障"
        - "耦合度持续增加"
      reason_rejected: "治标不治本"

    - name: "RabbitMQ 作为消息中间件"
      description: "使用 RabbitMQ 实现异步通信"
      pros:
        - "团队已有使用经验"
        - "支持多种消息模式"
      cons:
        - "吞吐量不如 Kafka"
        - "不支持消息回溯"
        - "集群运维复杂度高"
      reason_rejected: "吞吐量和消息回溯能力不满足需求"

    - name: "gRPC 流式通信"
      description: "使用 gRPC 双向流替代 REST"
      pros:
        - "性能优异"
        - "强类型契约"
      cons:
        - "本质仍是同步通信"
        - "不解决耦合问题"
      reason_rejected: "不解决核心的耦合和级联故障问题，但可作为同步场景的补充"

  consequences:
    positive:
      - "服务间解耦，新增消费者无需修改生产者"
      - "天然支持 CQRS 读写分离"
      - "Kafka 的消息回溯能力支持事件溯源和数据重建"
      - "系统弹性提升，单个服务故障不影响事件生产"
    negative:
      - "引入最终一致性，部分场景用户体验需要适配"
      - "团队需要学习 Kafka 运维和事件驱动开发模式"
      - "调试分布式事件流比同步调用更复杂"
      - "需要投入建设事件监控和追踪基础设施"
    risks:
      - "事件 Schema 演进管理不当可能导致消费者故障"
      - "消息积压可能导致延迟飙升"
      - "幂等处理实现不当可能导致数据重复"

  compliance:
    principles_aligned:
      - "AP-04: 事件作为一等公民"
      - "AP-06: 故障隔离"
      - "AP-01: 业务能力驱动"
    principles_violated: []

  impact:
    affected_entities:
      - "BC-ORDER"
      - "BC-INVENTORY"
      - "BC-PAYMENT"
      - "BC-NOTIFICATION"
      - "SVC-ORDER"
      - "SVC-INVENTORY"
      - "SVC-PAYMENT"
      - "SVC-NOTIFICATION"
    affected_teams:
      - "order-team"
      - "inventory-team"
      - "payment-team"
      - "platform-team"
    migration_needed: true
    estimated_effort: "3 个月渐进式迁移"

  reviews:
    - date: "2024-01-10"
      reviewer: "赵六"
      verdict: "needs-revision"
      comments: "需要补充 Schema 演进策略和消息积压处理方案"
    - date: "2024-01-12"
      reviewer: "赵六"
      verdict: "approved"
      comments: "补充内容完善，同意采纳"

  references:
    related_adrs:
      - "ADR-001: 微服务架构风格选择"
      - "ADR-007: 数据所有权与一致性策略"
    external_links:
      - "https://cloudevents.io/"
      - "https://microservices.io/patterns/data/saga.html"
```

## 5. ADR 简易示例（演示轻量级记录）

```yaml
# 06-governance/adrs/active/ADR-009-use-postgresql-for-new-services.yaml

adr:
  id: "ADR-009"
  title: "新服务默认使用 PostgreSQL"
  status: "accepted"
  date_proposed: "2024-02-01"
  date_decided: "2024-02-03"
  owner: "李四"

  tags:
    scope: "system-wide"
    category: "technology"
    impact_level: "medium"
    reversibility: "costly-reversible"

  context: |
    当前系统混用 MySQL、PostgreSQL、MongoDB，运维成本高。
    需要统一默认选型，降低认知和运维负担。

  decision: |
    新服务默认使用 PostgreSQL 15+。
    例外情况（如需要文档数据库）须提交 ADR 说明理由。

  alternatives:
    - name: "MySQL 8"
      pros: ["团队熟悉"]
      cons: ["JSON 支持弱于 PG", "扩展性不如 PG"]
      reason_rejected: "PostgreSQL 功能超集"

  consequences:
    positive:
      - "统一运维工具链"
      - "利用 PG 的 JSONB、全文搜索减少中间件依赖"
    negative:
      - "部分团队需要学习 PG 特性"

  compliance:
    principles_aligned: ["AP-09: 团队自治与对齐"]
```

```yaml
# 06-governance/adrs/active/ADR-012-api-versioning-strategy.yaml

adr:
  id: "ADR-012"
  title: "API 版本策略采用 URL 路径版本号"
  status: "accepted"
  date_proposed: "2024-02-15"
  date_decided: "2024-02-18"
  owner: "王五"

  tags:
    scope: "system-wide"
    category: "pattern"
    impact_level: "medium"
    reversibility: "easily-reversible"

  context: |
    多个团队对 API 版本管理方式不一致，有的用 Header，有的用 URL，
    导致 API Gateway 路由规则混乱。

  decision: |
    统一采用 URL 路径版本：`/api/v{N}/resource`。
    - 主版本号变更表示不兼容变更
    - 旧版本至少维护 6 个月
    - 通过 API Gateway 的 Sunset Header 通知废弃

  alternatives:
    - name: "Header 版本（Accept-Version）"
      pros: ["URL 更干净"]
      cons: ["调试不直观", "缓存不友好"]
      reason_rejected: "可发现性差，调试困难"

  consequences:
    positive: ["统一规范", "缓存友好", "调试直观"]
    negative: ["URL 中包含版本号不够 RESTful"]
```
