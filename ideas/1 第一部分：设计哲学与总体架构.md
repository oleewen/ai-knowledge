## 第一部分：设计哲学与总体架构

### 1.1 设计原则

本方案遵循以下核心设计原则：

| 原则 | 说明 | 实践指导 |
|------|------|----------|
| **关注点分离** | 四大视角独立建模，通过显式映射关联 | 每个视角有独立的元模型、目录和生命周期 |
| **单一事实源** | 每个知识片段只在一个地方维护，其他地方引用 | 应用级文档在代码仓库维护，系统级索引引用 |
| **渐进可发现** | 从任意视角进入，都能穿透到其他视角 | 通过交叉引用矩阵和映射注册表实现 |
| **机器可读优先** | 文档结构化为机器可解析格式，同时保证人类可读 | 采用 YAML/JSON frontmatter + Markdown 正文 |
| **演进式完整性** | 允许局部不完整，但要求不完整性可被追踪 | 通过覆盖率指标和待补全标记管理 |
| **联邦自治** | 应用级文档分散自治，系统级文档集中管控 | 通过契约规范和元数据索引桥接 |

```text
│                    知识消费层 (Knowledge Consumption Layer)           │
│  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ ┌────────────┐  │
│  │ 需求分析推导   │ │ 方案设计辅助   │ │ 影响分析引擎   │ │ 知识检索门户 │  │
│  └──────┬───────┘ └──────┬───────┘ └──────┬───────┘ └─────┬──────┘  │
├─────────┴────────────────┴────────────────┴───────────────┴─────────┤
│                    知识融合层 (Knowledge Fusion Layer)                │
│  ┌──────────────────────────────────────────────────────────────┐  │
│  │              交叉引用矩阵 (Cross-Reference Matrix)            │  │
│  │     业务↔产品 映射 │ 产品↔技术 映射 │ 技术↔数据 映射          │  │
│  │     业务↔数据 映射 │ 业务↔技术 映射 │ 产品↔数据 映射          │  │
│  └──────────────────────────────────────────────────────────────┘  │
├───────────────────────────────────────────────────────────────────┤
│                    知识建模层 (Knowledge Modeling Layer)             │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐          │
│  │ 业务视角  │  │ 产品视角  │  │ 技术视角  │  │ 数据视角  │          │
│  │ Business │  │ Product  │  │Technical │  │  Data    │          │
│  │  View    │  │  View    │  │  View    │  │  View    │          │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘  └────┬─────┘          │

│                    知识存储层 (Knowledge Storage Layer)              │
│  ┌────────────────────────┐  ┌────────────────────────────────┐   │
│  │   系统级文档库 (Hub)    │  │   应用级文档库 (Federated)      │   │

```

### 1.3 两级文档库的职责边界

```
┌─────────────────────────────────────────────────────────────┐
│                    系统级文档库 (Hub)                         │
│                                                             │
│  直接管理:                                                   │
│  ├── 全局业务域地图与限界上下文划分                             │
│  ├── 跨系统/跨应用的业务流程与集成关系                          │
│  ├── 系统级架构决策记录 (ADR)                                 │
│  ├── 四视角交叉映射注册表                                     │
│  ├── 全局数据字典与数据流向图                                  │
│  └── 产品线/产品模块的全局视图                                 │
│                                                             │
│  索引管理:                                                   │
│  ├── 各应用级文档的元数据索引                                  │
│  ├── 应用级 Specs 规约的摘要与引用                             │
│  └── 微服务接口的全局注册表                                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  应用级文档库 (Federated, per code repo)                     │
│                                                             │
│  直接管理:                                                   │
│  ├── 应用内部的模块/服务详细设计                               │
│  ├── BMAD/OpenSpec 生成的 Specs 规约                         │
│  ├── 应用内聚合根与领域对象的详细定义                           │
│  ├── API 接口规约 (OpenAPI/AsyncAPI)                         │
│  ├── 数据库 Schema 与迁移脚本                                 │
│  └── 应用级架构决策记录                                       │
│                                                             │
│  上报义务:                                                   │
│  ├── 按规范导出元数据清单 (manifest.yaml)                     │
│  └── 维护与系统级文档库的引用一致性                             │
└─────────────────────────────────────────────────────────────┘
```
