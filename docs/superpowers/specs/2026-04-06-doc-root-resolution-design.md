# doc_root 是否可由「执行 Skill / 目标工程 rules 上下文」决定

**日期**: 2026-04-06  
**状态**: 已实现（解析逻辑见 `scripts/sdx-doc-root.sh`；validate 脚本已接入）  
**范围**: 概念与约定层；可选后续在 `AGENTS.md` 等入口强化指针。

---

## 1. 术语

| 术语 | 含义（本设计） |
|------|----------------|
| **文档树根（doc_root）** | 相对**目标工程仓库根**的路径前缀，使得知识库、SDX 产物等落在 `{doc_root}/system/...` 或 `{doc_root}/application/...` 等约定子树下。中央库模板常用字面量 **`system/`**；`docs-init` 安装后会把 **`system/`** 改写为 **`docs_slash`**（如 `docs/`、`system/`）。 |
| **Skill 执行上下文** | Agent 运行时的工作区根、打开文件、用户传入参数、环境变量、以及 SKILL 步骤中的显式路径。 |
| **rules 下 md** | Cursor 等工具加载的规则文本；**本身不执行代码**，仅通过自然语言约束 Agent 行为。 |

---

## 2. 结论（可否）

| 来源 | 能否「决定」doc_root | 说明 |
|------|----------------------|------|
| **执行 Skill 时** | **可以** | 通过 **参数**（如现有 `--doc-root`）、**环境变量**、**SKILL 步骤内显式说明**（「先检测 `docs/` 与 `system/` 是否存在再选前缀」）或 **脚本 cwd** 组合解析，均可把 doc_root 绑定到当前工程。 |
| **仅依赖 rules 下 md** | **间接可以** | Rules **不能**像程序一样自动注入变量；但可在规则中约定：**必须先读取目标工程内某配置文件 / 元数据**，再按其中字段或目录存在性确定 doc_root。即「配置在工程里，规则要求 Agent 去读」。 |
| **与 docs-init 改写的关系** | 需对齐 | 安装后技能正文里的 **`system/`** 已被替换为 **`docs_slash`**；运行时「doc_root」若指「含 `system` 子目录的那一层」，应与该前缀一致，避免双重前缀错误。 |

---

## 3. 方案对比（如何统一「解析顺序」）

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **A（推荐）** | 约定解析顺序：**显式参数 / 环境变量** > **工程内单一配置文件**（可选，如未来 `doc-root.yaml` 或复用现有 meta）> **目录探测**（`docs/knowledge` vs `system/knowledge`）> **默认 `system`** | 可自动化、可审计；与现有 `validate-*.sh` 的 `--doc-root` 一致 | 需在 SKILL 或 rules 中写清顺序 |
| **B** | 仅 **环境变量**（如 `SDX_DOC_ROOT`），由开发者在 Cursor/CI 中设置 | 简单 | 易遗漏；跨机器需文档化 |
| **C** | 仅 **规则文字**要求「每次先问用户 doc_root」 | 无文件依赖 | 不可重复、不利于自动化 |

**推荐**：**A**；其中「工程内配置文件」若暂无，可先 **省略**，用 **目录探测 + 默认** 与 **参数** 补足。

---

## 4. rules 与 Skill 的分工（建议表述）

- **Rules**：声明「在本仓库/目标工程内编辑文档时，**doc_root 的解析须遵循** § 某节顺序；**禁止**在未确认前缀的情况下假设一律为 `system/`。」可指向 `AGENTS.md` 或本 spec 的链接（若入库）。
- **Skills**：在 `SKILL.md` 前置条件中增加一句：**若文档树非默认布局，须通过 `--doc-root` 或环境变量或工程内约定文件显式指定**；脚本类保持与现有 `DOC_ROOT` 默认 `system` 或 `.` 的兼容。

---

## 5. 与联邦 / docs-fetch 的边界

- **`docs-fetch` 的 `docs_root`**：来自 **`manifest`**，描述**远端目标工程**仓库内知识库根相对路径，与**本机工作区**内 Skill 使用的 doc_root **概念相关但来源不同**；设计时避免混用同名变量而不加注释。
