# 受众与文档语言（sdx-design）

主要读者为**架构师、研发负责人与实现工程师**；PRD 侧读者以业务与验收为主，见 [.agent/skills/sdx-prd/SKILL.md](../../sdx-prd/SKILL.md)。ADD **用技术语言**写清可实现方案，业务意图通过 **US-n / FR-n** 等编号追溯，避免在 ADD 中复述长篇产品叙事。

## 正文宜写 / 宜弱化

| 宜写入 ADD | 宜弱化或迁出 |
|------------|----------------|
| 服务边界、接口契约、错误码、幂等策略 | 用户故事全文（改为引用 PRD） |
| DDL、索引、缓存键与一致性策略 | 未经验证的实现猜测（标为待澄清） |
| Mermaid 架构/时序/状态/ER/类图 | 可执行源码与单测（归开发与 sdx-test） |
| DD-n 的备选方案与决策理由 | 与本轮 MVP 无关的扩展设计 |

## 与 PRD 的分工

PRD 描述「做什么、如何验收」；ADD 描述「怎么做、依赖什么系统与数据」。接口名、表结构、消息格式等**属于 ADD 与 specs/**；若 PRD 中已出现实现细节，设计时仍以 PRD 功能范围为约束，在 ADD 中给出**工程化**表述。
