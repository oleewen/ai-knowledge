# 技术设计质量验收清单（sdx-design）

步骤 4 验收用：Agent 在文档输出与评审时复制本清单并逐项勾选。原则与反模式见 [design-principles.md](design-principles.md)；模板结构见 [../assets/add-template.md](../assets/add-template.md)。

## 完整性

- [ ] 设计概述（第 1 章）：设计目标关联 PRD 与需求分析，设计约束完整
- [ ] 关键设计决策（1.3）：每个 DD-n 有决策点、结果、理由、备选方案
- [ ] 系统架构（2.1）：服务变更清晰，架构图 + 服务交互时序图完整
- [ ] 接口协议（2.2）：所有对外接口有名称、服务、能力、输入输出概要
- [ ] 领域模型（2.3）：聚合根/实体/值对象关系图完整，领域事件已列出
- [ ] 数据架构（2.4）：ER 图完整，分片与迁移方案（若适用）已描述
- [ ] 发布方案（2.5）：发布步骤、检查项、回滚方案齐全
- [ ] 应用架构（3.1）：容器级架构图完整，外部集成清晰
- [ ] API 详设（3.2）：每个 API-n 有签名、参数、响应、错误码、幂等性
- [ ] 业务逻辑（3.3）：核心类图、状态机、流程图/伪代码、一致性设计完整
- [ ] 数据访问（3.4）：DDL 完整，索引/分页/缓存策略明确
- [ ] 非功能性（3.5）：安全设计与可观测设计已覆盖
- [ ] 规约文件：按服务维度生成 API / 领域 / 数据 / 集成规约

## 一致性

- [ ] 架构设计与现有系统架构（`knowledge/technical/`）无冲突
- [ ] 领域模型与现有领域模型（`knowledge/business/`）一致
- [ ] API 设计遵循项目 API 命名规范
- [ ] 数据设计遵循项目数据库约束（表名前缀、`gmt_create`/`gmt_modified`）
- [ ] DD-n / API-n / LOGIC-n / TBL-n 编号连续无断号
- [ ] 规约文件与 ADD 设计条目双向对应
- [ ] 各章节内部交叉引用自洽（决策→设计→规约）

## 可行性

- [ ] 技术方案在现有基础设施上可实现
- [ ] 性能设计满足非功能需求（响应时间、吞吐量）
- [ ] 安全设计满足认证授权要求
- [ ] 发布方案可执行，回滚方案验证可行
- [ ] 禁止循环中 RPC/数据库调用（项目约束）

## 可追溯性

- [ ] 每个 DD-n 可追溯到 PRD 用户故事或 FR-n
- [ ] 每个 API-n 可追溯到 PRD 业务流程或用例
- [ ] 每个 TBL-n / DDL 可追溯到功能需求
- [ ] 每个规约文件标注关联 ADD 章节与 PRD 需求
- [ ] 非功能设计可追溯到需求分析中的非功能需求

## 格式规范

- [ ] 文末「## 文档元数据」内 fenced YAML 完整（id、title、version、status、created、updated、parent、mvp_phase）；**文件开头无** `---` YAML frontmatter
- [ ] 章节结构与 add-template.md 一致，编号连续
- [ ] Mermaid 图表可正确渲染（架构图、时序图、类图、状态机、ER 图）
- [ ] 表格格式规范，列宽合理
- [ ] 参考文档（第 4 章）列出所有规约文件路径
- [ ] 变更历史（第 4 章或附录约定节）记录初始版本
- [ ] 质量自查表（模板约定节）已逐项勾选
