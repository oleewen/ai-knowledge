# sdx-design 常见陷阱

---

## 输入与前置

**无 PRD 直接开始技术设计**：PRD 文档是硬输入；不存在时终止并提示先执行 `sdx-prd`。DD-n / API-n 无法追溯到 US-n / FR-n 时，可追溯性链条从根部断裂。

**忽略需求分析文档中的 MVP 边界**：严格聚焦 `--mvp` 参数指定的 MVP 范围；未包含的 FR-n 一律不纳入本次设计，否则引入不必要的抽象层或扩展点，过度设计。

**knowledge 目录缺失时不标注**：发出警告，在 §1 设计概述中显式标注「缺少知识库基线，以下架构设计仅基于 PRD 与 AGENTS.md」，不凭印象假设现有架构。

---

## 步骤 1：架构设计

**未引用现有架构直接假设系统结构**：步骤 1 必须先读 `knowledge/technical/` 和 INDEX_GUIDE.md §3，以现有架构为基线，只描述变更部分，否则设计与现有系统冲突，开发阶段才发现。

**领域模型与现有聚合冲突**：步骤 1 领域模型设计前必须读 `knowledge/business/`，优先扩展已有聚合；新增聚合须说明与现有聚合的边界区分，否则引入重复聚合，跨视角引用断链。

**关键设计决策无备选方案**：每个 DD-n 必须包含决策点、决策结果、决策理由、备选方案四个字段，否则后续评审或变更时无法理解决策背景，容易被推翻后重新设计。

**忽略 `--depth` 参数**：`quick` 仅设计系统架构 + 接口协议；`standard` 完整五维度；`deep` 额外增加性能建模、容量规划、灾备方案。

---

## 步骤 2：详细设计

**API 设计缺少幂等性说明**：每个 API-n 必须包含幂等性字段，说明保障方案（分布式锁键、业务唯一键等）；无需幂等的接口须显式标注原因，否则开发阶段各自实现，方案不一致。

**API 设计缺少错误码定义**：每个 API-n 必须定义错误码列表，包含错误码、错误信息、触发条件、HTTP 状态码，区分 BusinessException / SystemException / ValidationException，否则前端无法处理具体错误场景。

**数据表设计缺少 `gmt_create` / `gmt_modified`**：所有新增表必须包含这两个字段（`gmt_create` insert 时填充，`gmt_modified` insert/update 均填充），否则违反项目数据库约束，代码审查被打回。

**数据表设计无索引策略**：索引设计必须结合实际查询场景分析，说明每个索引对应的查询模式；避免冗余索引，否则上线后慢查询在生产环境才暴露。

**循环内调用 RPC 或数据库**：业务逻辑伪代码或流程图中禁止在 `for` 循环内调用 RPC 接口或执行数据库查询；批量操作须设计批量接口或批量查询，否则违反项目约束，N+1 问题导致性能瓶颈。

**忽略非功能性设计**：§3.5 必须覆盖安全设计（认证授权、数据脱敏、审计日志）和可观测设计（日志策略、监控报警规则、链路追踪），不能留空或写「参考通用方案」。

---

## 步骤 3：规约生成

**规约文件与 ADD 设计内容不一致**：规约文件必须从 ADD 对应章节直接提取，不得独立修改；ADD 变更时规约同步更新，否则开发者以规约为准实现，与 ADD 设计产生矛盾。

**规约文件缺少追溯标注**：每个规约文件头部必须标注 `source`（关联 ADD 章节，如 `ADD §3.2 API-001`）和 `requirement`（关联 PRD 需求，如 `FR-001`），否则规约与设计文档脱钩，无法追溯变更来源。

**规约目录结构不规范**：严格按 `specs/{service-name}/api/`、`domain/`、`data/`、`integration/` 四类目录组织，不平铺在 `specs/` 下，否则多服务场景下规约混乱，下游 sdx-test 按约定路径查找时失败。

---

## 步骤 4：文档输出

**跳过或重排模板章节**：章节结构与 `add-template.md` 完全一致；无内容的章节保留标题并标注「不适用」或「待补充」，不删除。下游 sdx-test 按固定章节引用，章节缺失或错位导致引用失效。

**文末元数据缺少 `mvp_phase` 或误用文件头 frontmatter**：勿在文件开头写 YAML frontmatter；仅在文末「## 文档元数据」fenced `yaml` 中填写字段；须包含 `id`、`title`、`version`、`status`、`created`、`updated`、`parent`、`mvp_phase`，初始 `status` 为 `draft`。

**在 ADD 中编写测试用例或实现代码**：ADD 只写伪代码和流程图描述逻辑，不写可执行代码；测试用例留给下游 sdx-test，否则 ADD 与代码实现耦合，代码变更时 ADD 难以同步维护。

**Mermaid 图表语法错误**：输出前验证 Mermaid 语法；时序图用 `sequenceDiagram`，状态机用 `stateDiagram-v2`，ER 图用 `erDiagram`，类图用 `classDiagram`，架构图用 `flowchart`。

**跳过质量门禁自查**：步骤 4 必须逐项检查 [reference/quality-checklist.md](reference/quality-checklist.md)，所有检查项通过后才输出文档。

## 快速自查清单

- [ ] PRD 文档已落盘且可用，`parent` 与 `mvp_phase` 字段已填写
- [ ] 设计严格聚焦目标 MVP 范围，未超越 MVP 边界
- [ ] 架构设计基于 `knowledge/technical/` 现有架构，仅描述变更部分
- [ ] 领域模型与 `knowledge/business/` 现有聚合无冲突
- [ ] 每个 DD-n 含决策点、结果、理由、备选方案
- [ ] 每个 API-n 含签名、参数、响应、错误码、幂等性说明
- [ ] 所有新增表含 `gmt_create` / `gmt_modified`，有索引策略
- [ ] 业务逻辑中无循环内 RPC / 数据库调用
- [ ] §3.5 非功能性设计已覆盖安全与可观测性
- [ ] 规约文件按 `{service-name}/{type}/` 目录结构组织
- [ ] 每个规约文件头部含 `source` 和 `requirement` 追溯标注
- [ ] 规约内容与 ADD 对应章节一致
- [ ] 五章结构完整，无删除章节
- [ ] 文末元数据 YAML 字段完整（含 `mvp_phase`）；文件开头无 `---` YAML
- [ ] ADD 中无可执行代码或测试用例
- [ ] 所有 Mermaid 图表语法正确可渲染
- [ ] quality-checklist.md 已逐项勾选
