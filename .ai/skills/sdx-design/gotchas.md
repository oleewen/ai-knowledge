# sdx-design 常见陷阱（Gotchas）

本文记录执行 `sdx-design` Skill 时高频踩坑点，供 Agent 与人类开发者参考。

---

## 1. 输入与前置条件

### 1.1 无 PRD 直接开始技术设计
**陷阱**：用户描述功能需求，Agent 跳过 `sdx-prd` 直接输出 ADD。  
**后果**：缺少用户故事（US-n）和功能需求（FR-n）作为设计追溯锚点，DD-n / API-n 无法追溯到业务需求。  
**正确做法**：PRD 文档是硬输入；不存在时终止并提示先执行 `sdx-prd`。

### 1.2 忽略需求分析文档中的 MVP 边界
**陷阱**：设计时未限定目标 MVP 范围，将后续 MVP 的功能一并纳入设计。  
**后果**：ADD 超越 MVP 边界，引入不必要的抽象层或扩展点，过度设计。  
**正确做法**：严格聚焦 `--mvp` 参数指定的 MVP 范围；需求分析文档中该 MVP 未包含的 FR-n 一律不纳入本次设计。

### 1.3 knowledge 目录缺失时不标注
**陷阱**：`knowledge/technical/` 或 `knowledge/business/` 不存在，Agent 凭印象假设现有架构与领域模型。  
**后果**：架构设计与实际系统不一致，领域模型与现有聚合冲突，产出含幻觉设计。  
**正确做法**：发出警告，在 §1 设计概述中显式标注「缺少知识库基线，以下架构设计仅基于 PRD 与 AGENTS.md」。

---

## 2. 步骤 1：架构设计

### 2.1 未引用现有架构直接假设系统结构
**陷阱**：未读 `knowledge/technical/` 和 INDEX_GUIDE.md §3，直接凭经验描述服务调用关系。  
**后果**：设计与现有系统架构冲突，开发阶段才发现，返工成本高。  
**正确做法**：步骤 1 必须先读 `knowledge/technical/` 和 INDEX_GUIDE.md §3，以现有架构为基线，只描述变更部分。

### 2.2 领域模型与现有聚合冲突
**陷阱**：设计新聚合时未查阅 `knowledge/business/`，与已有 AGG-* 边界重叠或命名冲突。  
**后果**：引入重复聚合，跨视角引用断链，`knowledge-build` 下次运行时产生 ID 冲突。  
**正确做法**：步骤 1 领域模型设计前必须读 `knowledge/business/`，优先扩展已有聚合，新增聚合须说明与现有聚合的边界区分。

### 2.3 关键设计决策无备选方案
**陷阱**：DD-n 只记录「决策结果」，不记录备选方案与决策理由。  
**后果**：后续评审或变更时无法理解决策背景，容易被推翻后重新设计。  
**正确做法**：每个 DD-n 必须包含决策点、决策结果、决策理由、备选方案四个字段。

### 2.4 忽略 `--depth` 参数调整设计粒度
**陷阱**：无论用户传入 `quick` 还是 `deep`，始终执行完整五维度架构设计。  
**后果**：`quick` 场景下耗时过长；`deep` 场景下缺少性能建模与容量规划。  
**正确做法**：`quick` 仅设计系统架构 + 接口协议；`standard` 完整五维度；`deep` 额外增加性能建模、容量规划、灾备方案。

---

## 3. 步骤 2：详细设计

### 3.1 API 设计缺少幂等性说明
**陷阱**：API 详设只有签名和参数，未说明幂等性保障方案。  
**后果**：开发阶段各自实现幂等逻辑，方案不一致；重复提交场景产生数据问题。  
**正确做法**：每个 API-n 必须包含幂等性字段，说明保障方案（分布式锁键、业务唯一键等）；无需幂等的接口须显式标注原因。

### 3.2 API 设计缺少错误码定义
**陷阱**：API 响应只定义成功结构，错误码留空或写「参考通用错误码」。  
**后果**：前端/调用方无法处理具体错误场景；测试设计无法覆盖异常路径。  
**正确做法**：每个 API-n 必须定义错误码列表，包含错误码、错误信息、触发条件、HTTP 状态码，区分 BusinessException / SystemException / ValidationException。

### 3.3 数据表设计缺少 `gmt_create` / `gmt_modified`
**陷阱**：DDL 中未包含 `gmt_create` 和 `gmt_modified` 字段。  
**后果**：违反项目数据库约束，代码审查时被打回，DDL 需重新设计。  
**正确做法**：所有新增表必须包含 `gmt_create`（insert 时填充）和 `gmt_modified`（insert/update 均填充）。

### 3.4 数据表设计无索引策略
**陷阱**：DDL 只有字段定义，未分析查询场景，未设计索引。  
**后果**：上线后慢查询，性能问题在生产环境才暴露。  
**正确做法**：索引设计必须结合实际查询场景分析，说明每个索引对应的查询模式；避免冗余索引。

### 3.5 循环内调用 RPC 或数据库
**陷阱**：业务逻辑伪代码或流程图中出现 `for` 循环内调用 RPC 接口或执行数据库查询。  
**后果**：违反项目约束，N+1 问题导致性能瓶颈，代码审查被打回。  
**正确做法**：批量操作须设计批量接口或批量查询；循环内操作改为先批量获取数据再循环处理。

### 3.6 忽略非功能性设计
**陷阱**：详细设计只覆盖功能实现，§3.5 非功能性设计留空或写「参考通用方案」。  
**后果**：安全漏洞（如缺少鉴权）、可观测性缺失（如无监控告警）在上线后才发现。  
**正确做法**：§3.5 必须覆盖安全设计（认证授权、数据脱敏、审计日志）和可观测设计（日志策略、监控报警规则、链路追踪）。

---

## 4. 步骤 3：规约生成

### 4.1 规约文件与 ADD 设计内容不一致
**陷阱**：规约文件从 ADD 提取后独立修改，与 ADD 中的 API 签名或 DDL 产生偏差。  
**后果**：开发者以规约为准实现，与 ADD 设计不一致，评审时发现矛盾。  
**正确做法**：规约文件必须从 ADD 对应章节直接提取，不得独立修改；ADD 变更时规约同步更新。

### 4.2 规约文件缺少追溯标注
**陷阱**：规约 YAML 文件头部未标注关联的 ADD 章节和 PRD 需求。  
**后果**：规约与设计文档脱钩，无法追溯变更来源，维护时不知道修改规约是否需要同步 ADD。  
**正确做法**：每个规约文件头部必须标注 `source`（关联 ADD 章节，如 `ADD §3.2 API-001`）和 `requirement`（关联 PRD 需求，如 `FR-001`）。

### 4.3 规约目录结构不规范
**陷阱**：所有规约文件平铺在 `specs/` 下，未按 `{service-name}/{type}/` 分层。  
**后果**：多服务场景下规约混乱，无法按服务维度管理；下游 `sdx-test` 按约定路径查找规约时失败。  
**正确做法**：严格按 `specs/{service-name}/api/`、`domain/`、`data/`、`integration/` 四类目录组织。

---

## 5. 步骤 4：文档输出

### 5.1 跳过或重排模板章节
**陷阱**：认为某章节「不适用」直接删除，或调整章节顺序。  
**后果**：下游 `sdx-test` 按固定章节引用 ADD，章节缺失或错位导致引用失效。  
**正确做法**：章节结构与 `add-template.md` 完全一致；无内容的章节保留标题并标注「不适用」或「待补充」。

### 5.2 frontmatter 缺少 `mvp_phase` 字段
**陷阱**：frontmatter 只填写通用字段，遗漏 `mvp_phase`。  
**后果**：无法从文档元数据判断本 ADD 对应哪个 MVP 阶段，多 MVP 场景下管理混乱。  
**正确做法**：frontmatter 必须包含 `id`、`title`、`version`、`status`、`created`、`updated`、`parent`、`mvp_phase` 八个字段。

### 5.3 在 ADD 中编写测试用例或实现代码
**陷阱**：在业务逻辑设计（§3.3）中写出完整 Java 实现代码，或在 §3.2 中写出测试用例。  
**后果**：范围蔓延，ADD 与代码实现耦合，代码变更时 ADD 难以同步维护。  
**正确做法**：ADD 只写伪代码和流程图描述逻辑，不写可执行代码；测试用例留给下游 `sdx-test`。

### 5.4 Mermaid 图表语法错误
**陷阱**：架构图、时序图、类图、状态机、ER 图使用了不合法的 Mermaid 语法，无法渲染。  
**后果**：文档在 GitHub / IDE 中显示为原始文本，可视化失效，评审时无法直观理解设计。  
**正确做法**：输出前验证 Mermaid 语法；时序图用 `sequenceDiagram`，状态机用 `stateDiagram-v2`，ER 图用 `erDiagram`，类图用 `classDiagram`。

### 5.5 跳过质量门禁自查
**陷阱**：ADD 写完后直接输出，未逐项检查 `reference/quality-checklist.md`。  
**后果**：编号断号、规约脱钩、追溯链断裂等问题流入下游，被 `sdx-test` 或代码评审发现时返工成本高。  
**正确做法**：步骤 4 必须执行质量门禁自查，所有检查项通过后才输出文档。

## 快速检查清单

执行完毕后，对照以下项目快速自查：

- [ ] PRD 文档已落盘且可用，`parent` 字段已填写
- [ ] 设计严格聚焦目标 MVP 范围，未超越 MVP 边界
- [ ] 架构设计基于 `knowledge/technical/` 现有架构，仅描述变更部分
- [ ] 领域模型与 `knowledge/business/` 现有聚合无冲突
- [ ] 每个 DD-n 含决策点、结果、理由、备选方案
- [ ] 每个 API-n 含签名、参数、响应、错误码、幂等性说明
- [ ] 所有新增表含 `gmt_create` / `gmt_modified`，有索引策略
- [ ] 业务逻辑中无循环内 RPC / 数据库调用
- [ ] §3.5 非功能性设计已覆盖安全与可观测性
- [ ] 规约文件按 `{service-name}/{type}/` 目录结构组织
- [ ] 每个规约文件头部含 `source` 和 `requirement` 追溯标注
- [ ] 规约内容与 ADD 对应章节一致
- [ ] 章节结构与 `add-template.md` 完全一致，无删除章节
- [ ] frontmatter 八个字段完整，含 `mvp_phase`
- [ ] ADD 中无可执行代码或测试用例
- [ ] 所有 Mermaid 图表语法正确可渲染
- [ ] 质量门禁清单已逐项勾选
