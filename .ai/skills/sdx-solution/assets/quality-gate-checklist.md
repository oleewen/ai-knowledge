# 解决方案质量门禁清单

步骤 5 验收模板。Agent 在执行文档输出与评审时复制本清单并逐项勾选。

## 完整性

- [ ] 业务背景（§1）：需求来源、现状、问题/痛点均已描述
- [ ] 业务目标（§2）：每个 G-n 有度量指标和目标值
- [ ] 核心场景（§3.1）：覆盖主要业务场景
- [ ] 用户角色（§3.2）：列出所有涉及角色及关注点
- [ ] 约束条件（§3.3）：业务/技术/资源约束完整
- [ ] 待澄清项（§3.5）：歧义/缺失信息已标注为 Q-n
- [ ] 影响面（§4）：覆盖所有受影响服务/模块，传播路径清晰
- [ ] 冲突分析（§5）：业务与系统冲突完整覆盖（或已标注 `--skip-conflict`）
- [ ] 解决思路（§6.1）：整体策略明确
- [ ] 关键决策（§6.2）：每项有推荐方案与理由
- [ ] 范围界定（§6.4）：In Scope / Out of Scope 清晰
- [ ] 可行性（§7）：技术可行性、资源评估、风险 R-n 完整
- [ ] MVP 拆分（§8）：每个 MVP 有独立业务价值

## 一致性

- [ ] 需求描述与 `knowledge/` 现有文档无矛盾
- [ ] 影响面与架构文档（INDEX_GUIDE.md）一致
- [ ] 文档内部 G-n / C-n / R-n 编号无重复、无断号
- [ ] 各章节交叉引用自洽（目标→需求→方案→风险）

## 可行性

- [ ] 技术方案在现有架构下可实现
- [ ] 资源估算合理（时间、人力、基础设施）
- [ ] 每个 R-n 有对应的应对策略

## 可追溯性

- [ ] 每个 G-n 可追溯到原始业务描述
- [ ] 每个受影响功能可追溯到具体服务/模块
- [ ] 每个 C-n / C-Tn 可追溯到具体规则或契约
- [ ] 每个 R-n 可追溯到冲突分析或可行性评估

## 格式规范

- [ ] frontmatter 完整（id、title、version、status、created、updated）
- [ ] 章节结构与 solution-template.md 一致，编号连续
- [ ] 表格格式规范，列宽合理
- [ ] 术语表（§9.1）涵盖本文档专用术语
- [ ] 参考文档（§9.2）列出引用的知识库/规约文件
