# sdx-solution 常见陷阱（Gotchas）

本文记录执行 `sdx-solution` Skill 时高频踩坑点，供 Agent 与人类开发者参考。

---

## 1. 输入与前置条件

### 1.1 无原始业务描述直接开始制定方案
**陷阱**：用户只给出一句话标题（如「做一个审批功能」），Agent 直接展开方案制定，未要求提供原始业务描述。  
**后果**：方案缺乏业务背景与动机支撑，业务目标（G-n）无法追溯到原始需求，可追溯性链条从根部断裂。  
**正确做法**：至少一种原始来源（邮件、会议纪要、工单等）是硬输入；描述过于简短时暂停并请求用户补充背景与期望价值。

### 1.2 通读全仓知识库再开始分析
**陷阱**：步骤 1 打开 `knowledge/**` 下所有文件进行「全面了解」。  
**后果**：上下文爆炸，耗时极长，且大量内容与本次业务需求无关。  
**正确做法**：从业务描述中识别涉及的领域与服务，按图索骥——只打开与本次需求直接相关的知识库视角文件。

### 1.3 knowledge 目录缺失时不标注
**陷阱**：`knowledge/` 目录不存在，Agent 凭印象假设现有系统状态，未在文档中说明。  
**后果**：影响面评估与冲突分析基于幻觉，读者误以为已参照知识库基线，实际上系统边界未经验证。  
**正确做法**：在文档中显式标注「缺少知识库基线，以下影响面评估仅基于业务描述，存在分析盲区」。

---

## 2. 步骤 1：需求提取与结构化

### 2.1 吞没歧义自行补全
**陷阱**：业务描述中某需求模糊（如「支持灵活配置」），Agent 自行假设具体含义后继续提取。  
**后果**：结构化需求基于错误假设，下游 `sdx-analysis` 全部偏离，评审时才发现，返工成本高。  
**正确做法**：歧义项标注为「待澄清」，在文档中显式记录疑问点；可基于已有内容继续，但须在文档中标注假设前提。

### 2.2 将技术方案混入需求提取
**陷阱**：步骤 1 提取需求时直接写出技术实现方案（如「使用消息队列解耦」「引入缓存层」）。  
**后果**：需求与方案混淆，步骤 4 方案制定失去独立性，过早锁定技术选型，备选方案评估缺失。  
**正确做法**：步骤 1 只提取「做什么」和「为什么」，不涉及「怎么做」；技术方案留给步骤 4。

### 2.3 未区分新增/变更/修复类型
**陷阱**：所有需求统一列为「功能需求」，未区分是新增功能、变更现有逻辑还是修复缺陷。  
**后果**：影响面评估无法准确判断变更范围，冲突分析遗漏现有逻辑的兼容性问题。  
**正确做法**：每条需求须标注类型（新增 / 变更 / 修复）；变更类需求须额外标注被变更的现有功能或规则。

### 2.4 优先级缺失或全部标为高优先级
**陷阱**：所有需求未标注优先级，或全部标为「高优先级」。  
**后果**：步骤 4 MVP 拆分建议无依据，下游 `sdx-analysis` 无法判断哪些需求必须进入首个 MVP。  
**正确做法**：每条需求须标注优先级（P0–P3）；优先级无法确定时暂停并请求用户确认，不得全部默认为高。

---

## 3. 步骤 2：影响面评估与分析

### 3.1 影响面评估停留在功能层面，遗漏数据与接口
**陷阱**：影响面只列出受影响的功能模块，未评估数据模型变更、接口契约变更、下游消费方影响。  
**后果**：数据迁移成本、接口兼容性问题在开发阶段才暴露，进度风险被严重低估。  
**正确做法**：影响面评估须覆盖四个维度：功能影响、数据影响（表结构/字段变更）、接口影响（契约变更/新增）、下游影响（消费方/依赖方）。

### 3.2 影响传播路径未标注直接/间接层级
**陷阱**：影响面列表平铺所有受影响项，未区分直接影响与间接影响。  
**后果**：评审时无法判断影响范围的优先级，直接影响与间接影响混淆，资源评估失准。  
**正确做法**：影响传播路径须标注层级：直接影响（本次变更直接修改的模块/接口）→ 间接影响（因直接影响而被动受影响的下游）。

### 3.3 `--depth=quick` 时跳过影响面评估
**陷阱**：`--depth=quick` 时完全省略步骤 2，未将影响面合并入步骤 4。  
**后果**：解决方案文档缺少影响面章节，下游 `sdx-analysis` 无法基于影响面进行 MVP 拆分。  
**正确做法**：`--depth=quick` 时将影响面评估简化后合并入步骤 4，不得完全省略；至少列出高影响项。

---

## 4. 步骤 3：冲突识别与化解

### 4.1 `--skip-conflict` 滥用于有已有系统的场景
**陷阱**：系统中已有相关功能，但用户传入 `--skip-conflict=true`，Agent 直接跳过冲突分析。  
**后果**：与现有业务规则、进行中需求的冲突未被识别，开发阶段才发现，返工成本高。  
**正确做法**：`--skip-conflict` 仅适用于全新系统或确认无已有逻辑的纯新增场景；存在已有系统时须执行冲突分析，即使用户传入该参数也须发出警告。

### 4.2 冲突识别只覆盖业务冲突，遗漏系统冲突
**陷阱**：冲突分析只列出业务规则冲突（如「新规则与现有审批流程矛盾」），未分析系统层面的模型冲突、契约冲突、资源竞争。  
**后果**：系统冲突（如数据库表字段语义冲突、接口版本兼容性问题）在设计阶段才暴露。  
**正确做法**：冲突识别须覆盖两类：业务冲突（规则/流程/数据语义）与系统冲突（模型边界/接口契约/资源竞争）。

### 4.3 冲突化解方案缺少成本与风险评估
**陷阱**：每项冲突只给出化解方案，未评估化解成本（工作量/时间）与残余风险。  
**后果**：评审时无法判断化解方案的可行性，高成本化解方案被默认接受，资源规划失准。  
**正确做法**：每项冲突的化解方案须包含：化解策略、预估成本（高/中/低）、残余风险说明；高成本化解方案须提供备选策略。

---

## 5. 步骤 4：方案制定与评估

### 5.1 业务目标不可度量
**陷阱**：业务目标写「提升用户体验」「优化审批效率」，无具体度量指标。  
**后果**：验收标准无法度量，上线后无法判断目标是否达成，业务价值无法量化。  
**正确做法**：每个业务目标（G-n）须包含可度量的成功指标（如「审批平均时长从 2 天降至 4 小时」）；无法量化时标注「待澄清」。

### 5.2 关键决策无备选方案对比
**陷阱**：方案制定只给出最终选择，未记录备选方案与决策理由。  
**后果**：评审时无法理解决策背景，容易被推翻后重新制定，且相同决策在下游重复讨论。  
**正确做法**：每个关键决策须包含：决策点、选定方案、备选方案、选定理由；备选方案须说明未选原因。

### 5.3 MVP 拆分建议按功能类型而非业务价值
**陷阱**：MVP 拆分按「查询功能」「写入功能」「配置功能」等技术类型划分，而非按可独立交付的业务价值。  
**后果**：每个 MVP 无法独立演示给业务方，交付价值不可见，下游 `sdx-analysis` 拆分结果不可靠。  
**正确做法**：每个 MVP 必须能独立演示给业务方并解决具体业务问题；拆分后自问「这个 MVP 能解决什么业务问题」，答不上来则重拆。

### 5.4 可行性评估流于形式
**陷阱**：技术可行性评估只写「技术上可行」，未分析具体技术风险点与资源约束。  
**后果**：隐性技术风险（如第三方依赖不稳定、团队技能缺口）在开发阶段才暴露，进度风险被低估。  
**正确做法**：可行性评估须覆盖：技术风险点（含概率与影响）、资源约束（人力/时间/基础设施）、外部依赖 SLA；高风险项须有应对预案。

## 6. 步骤 5：文档输出与评审

### 6.1 跳过或重排模板章节
**陷阱**：认为某章节「不适用」直接删除，或调整章节顺序以「更符合逻辑」。  
**后果**：下游 `sdx-analysis` 按固定章节引用解决方案文档，章节缺失或错位导致引用失效，追溯链断裂。  
**正确做法**：章节结构与 `solution-template.md` 九章结构完全一致；无内容的章节保留标题并标注「不适用」或「待补充」，不得删除。

### 6.2 frontmatter 字段缺失
**陷阱**：输出文档缺少 `status` 或 `id` 字段，或 `id` 格式不符合 `YYYYMMDD-SEQ` 规范。  
**后果**：文档管理混乱，下游 `sdx-analysis` 的 `parent` 字段无法正确引用，追溯链断裂。  
**正确做法**：frontmatter 必须包含 `id`、`title`、`version`、`status`、`created`、`updated` 六个字段，初始 `status` 为 `draft`；`id` 格式严格遵循 `YYYYMMDD-SEQ`。

### 6.3 在解决方案文档中写入 PRD 或 ADD 级内容
**陷阱**：在方案制定章节中写出用户故事（As a... I want...）或技术实现细节（如「使用 Kafka 实现异步解耦」）。  
**后果**：范围蔓延，解决方案与 PRD / ADD 职责混淆，下游 Skill 无法在此基础上增量工作。  
**正确做法**：解决方案文档只描述「业务目标」「解决思路」「影响面」「冲突化解」「MVP 拆分建议」，不涉及用户故事、用例、技术选型细节。

### 6.4 跳过质量门禁自查
**陷阱**：文档写完后直接输出，未逐项检查 `quality-gate-checklist.md`。  
**后果**：业务目标不可度量、影响面遗漏、追溯链断裂等问题流入下游，被 `sdx-analysis` 或评审发现时返工成本高。  
**正确做法**：步骤 5 必须执行质量门禁自查，所有检查项通过后才输出文档。

---

## 快速检查清单

执行完毕后，对照以下项目快速自查：

- [ ] 原始业务描述已作为输入，`id` 字段格式符合 `YYYYMMDD-SEQ`
- [ ] 知识库缺失时已在文档中显式标注分析盲区
- [ ] 需求提取已区分新增/变更/修复类型
- [ ] 每条需求已标注优先级（P0–P3），无全部默认高优先级
- [ ] 歧义项已标注「待澄清」，未自行假设
- [ ] 影响面覆盖功能、数据、接口、下游四个维度
- [ ] 影响传播路径已标注直接/间接层级
- [ ] 冲突识别覆盖业务冲突与系统冲突两类
- [ ] 每项冲突化解方案含成本与残余风险评估
- [ ] 每个业务目标（G-n）含可度量成功指标
- [ ] 关键决策含备选方案与选定理由
- [ ] MVP 拆分按业务价值划分，每个 MVP 可独立演示
- [ ] 可行性评估覆盖技术风险、资源约束、外部依赖
- [ ] 章节结构与 `solution-template.md` 九章结构完全一致，无删除章节
- [ ] frontmatter 六个字段完整，`status` 初始为 `draft`
- [ ] 文档中无用户故事、用例、技术选型细节
- [ ] 质量门禁清单已逐项勾选
