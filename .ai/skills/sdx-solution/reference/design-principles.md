# 设计原则与反模式

sdx-solution 技能的完整设计约束。主文件 SKILL.md 中的「核心约束」为精简版，本文件为完整规范。

---

## 设计原则

### 1. 模板驱动

输出严格遵循 [../assets/solution-template.md](../assets/solution-template.md) 的九章结构：

```
§1 业务背景 → §2 业务目标 → §3 需求概述 → §4 影响面评估
→ §5 冲突分析 → §6 解决方案 → §7 可行性评估 → §8 MVP拆分建议 → §9 附录
```

章节顺序与编号不可调整；无内容的章节保留标题并标注「不适用」或「待补充」，不得删除。

### 2. 业务可读

正文使用业务语言，面向业务方与产品可读。完整语言规范见 [audience-and-language.md](audience-and-language.md)。

核心原则：用「谁能做什么」「流程如何变」「对外承诺是什么」代替系统内部构造。

### 3. 证据优先

影响面评估与冲突分析须引用 `knowledge/`、AGENTS.md 或工程事实，禁止凭空臆测。

| 证据类型 | 格式 | 示例 |
|----------|------|------|
| 知识库实体 | `{视角}-{ID} {名称}` | `BC-001 FeeAppealContext` |
| 文档章节 | `{文件} §{章节}` | `INDEX_GUIDE.md §3.2` |
| 代码位置 | `{类}:{方法}` | `PolicyAppealApi:create` |
| 配置文件 | `{路径}:{键}` | `application.yml:spring.datasource` |

证据仅用于 Agent 内部分析校准，**写入文档时必须转写为业务表述**。

### 4. 按需加载

仅在本轮任务需要时打开文件：

- 步骤 1：仅读原始业务描述
- 步骤 2：按影响面打开 `knowledge/` 对应视角
- 步骤 3：打开相关的现有规约与架构文档
- 步骤 4：综合已读内容制定方案

禁止为「完整性」通读 `knowledge/**` 或全仓源码。

### 5. 歧义标注

对业务描述中的歧义、矛盾、信息缺失，标注为待澄清项（Q-n），填入文档 §3.5。禁止自行假设缺失信息后继续，应暂停并请求用户澄清。

**决策阈值**：歧义项 > 3 且涉及核心目标 → 暂停，输出已提取结构化信息与待澄清清单；歧义项 ≤ 3 且不涉及核心目标 → 继续，在 §3.5 标注后进入下一步。

### 6. 范围边界

本技能仅负责输出解决方案文档。不产出 PRD、ADD、测试设计或代码——那些是下游 sdx-analysis / sdx-prd / sdx-design / sdx-test 的职责。

### 7. 可追溯性

| 追溯链 | 要求 |
|--------|------|
| G-n → 原始业务描述 | 每个业务目标可追溯到原始需求中的具体段落 |
| 影响点 → 业务能力/协作环节 | 每个受影响项可追溯到具体业务能力（非系统模块名） |
| C-n / C-Tn → 具体规则或契约 | 每个冲突可追溯到具体的业务规则或协作约定 |
| R-n → 冲突分析或可行性评估 | 每个风险可追溯到其来源分析 |

### 8. ID 编号规范

| 前缀 | 用途 | 示例 |
|------|------|------|
| `SOLUTION-{YYYYMMDD}-{SEQ}` | 文档编号 | `SOLUTION-20260327-001` |
| `G-{N}` | 业务目标编号 | `G-1`、`G-2` |
| `Q-{N}` | 待澄清问题编号 | `Q-1`、`Q-2` |
| `C-{N}` | 业务冲突编号 | `C-1`、`C-2` |
| `C-T{N}` | 系统冲突编号 | `C-T1`、`C-T2` |
| `R-{N}` | 风险编号 | `R-1`、`R-2` |

编号在文档内连续、无重复、无断号。

## 反模式清单（禁止）

| 反模式 | 说明 |
|--------|------|
| 技术语言入正文 | 在解决方案正文中写出接口名、表名、中间件、框架等技术词 |
| 臆测影响面 | 未引用 knowledge 或工程事实即声称某功能受影响 |
| 吞没歧义 | 将不明确需求自行补全后继续，不标注待澄清 |
| 范围蔓延 | 在解决方案中直接写出 PRD 级别的用户故事或技术设计 |
| 模板跳章 | 跳过或重排 solution-template.md 的章节结构 |
| 通读全库 | 一次性读取全仓知识库而不考虑本轮任务实际需要 |
| 无编号引用 | 目标/冲突/风险无统一编号，导致不可追溯 |
| 单方案定论 | 有多种可行方案时不做对比，直接给出唯一方案 |
| MVP 硬拆 | 不考虑业务价值独立性，机械按功能维度拆分 MVP |
| 空章提交 | 章节标题下无任何内容且未标注「不适用」或「待补充」 |
| skip-conflict 滥用 | 存在已有系统时仍使用 `--skip-conflict`，跳过必要的冲突分析 |

## 质量标准

### 完整性

- 业务背景与动机清晰描述
- 所有业务目标（G-n）可度量（有指标和目标值）
- 影响面覆盖功能、数据、接口、下游四个维度
- 冲突分析覆盖业务与系统两个维度
- 解决思路明确，关键决策有备选方案与理由
- 范围边界清晰（In Scope / Out of Scope）

### 一致性

- 需求描述与现有系统文档（knowledge/、AGENTS.md）无矛盾
- 影响面与架构文档（INDEX_GUIDE.md）一致
- 文档内部各章节交叉引用一致（目标→需求→方案→风险）
- G-n / C-n / R-n 编号无重复、无断号

### 可行性

- 技术方案在现有架构下可实现（以业务语言表述）
- 资源估算合理（时间、人力、协作依赖）
- 每个 R-n 有对应的应对策略

### 可追溯性

- G-n → 原始业务描述
- 受影响功能 → 具体业务能力/协作环节
- C-n / C-Tn → 具体规则或契约
- R-n → 冲突分析或可行性评估

## 最佳实践

### 需求提取

- 从多个来源交叉验证关键需求点
- 对数字指标（时间、量级、频率）显式确认
- 将隐含需求（安全、性能、兼容）显式化为非功能需求
- 每条需求标注类型（新增/变更/修复）和优先级（P0–P3）

### 影响面评估

- 从直接影响出发，沿业务流程/协作链追踪间接影响
- 标注影响程度时考虑用户感知、数据完整性、业务连续性三个维度
- `--depth=quick` 时至少列出高影响项，不得完全省略

### 冲突化解

- 对每个冲突提供至少一个化解方案
- 高严重度冲突提供备选化解方案
- 评估化解成本时区分一次性成本与持续成本

### MVP 拆分

- 每个 MVP 具备独立的业务交付价值，可独立演示给业务方
- MVP 间依赖单向（MVP-N+1 可依赖 MVP-N，反向禁止）
- 核心/高价值功能优先
- 自问「这个 MVP 能解决什么业务问题」，答不上来则重拆

## 错误处理

| 错误场景 | 处理方式 |
|----------|----------|
| 业务描述过于模糊 | 暂停，输出已提取的结构化信息与待澄清清单，请求用户补充 |
| knowledge 目录缺失 | 发出警告，仅基于业务描述和 AGENTS.md 完成分析，在文档中标注「缺少知识库基线，影响面评估存在分析盲区」 |
| 模板文件不存在 | 终止，提示创建 `.ai/skills/sdx-solution/assets/solution-template.md` |
| 冲突无法化解 | 记录为高风险（R-n），标注需升级决策，列出各方案利弊 |
| 输出目录不存在 | 自动创建 `system/solutions/` 目录 |
