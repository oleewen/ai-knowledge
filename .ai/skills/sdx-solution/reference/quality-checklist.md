# 解决方案质量验收清单（sdx-solution）

步骤 5 验收用：Agent 在文档输出与评审时复制本清单并逐项勾选，**所有项通过后才输出文档**。原则与反模式见 [design-principles.md](design-principles.md)；模板结构见 [../assets/solution-template.md](../assets/solution-template.md)。

---

## 一、完整性

- [ ] **§1.1–§1.4 背景与目标**：现状、问题、业务目标（G-n 可度量）、业务价值均已覆盖
- [ ] **§2.1–§2.4 范围与约束**：核心场景、角色、范围边界（含 In/Out/成功标准）、关键约束（含交付节点）完整
- [ ] **§3.1–§3.3 影响**：叙述性影响面、§3.2 表、§3.3 传播路径齐备；覆盖功能、数据、对外承诺、下游四维度
- [ ] **§3.4 业务冲突**：冲突表完整（或已标注 `--skip-conflict` 且场景合法）；每项含化解成本；协作/契约类冲突以业务后果表述
- [ ] **§4.1 解决思路**：整体策略明确（可有图示占位说明）
- [ ] **§4.2 方案对比**：存在多方案时已填对比表；仅单一路径时可标注「不适用」
- [ ] **§4.3 关键决策**：每项有推荐方案、备选与选定理由
- [ ] **§5.1 风险评估**：每个 R-n 有应对策略
- [ ] **§5.2 待澄清问题**：Q-n 无自行假设；列含影响范围、责任人、期望澄清日期、状态等（与模板表头一致）
- [ ] **§6.1 MVP 拆分**：每个 MVP 有独立业务价值，可独立演示给业务方
- [ ] **§6.2 关键里程碑**：关键节点与验收标准有落位（无则标注「待补充」）
- [ ] **§7.1–§7.3 附录**：术语与参考齐备；§7.3 按需填写

---

## 二、业务语言

- [ ] 正文无具体技术栈名称（Spring Boot、Kafka、Redis、TiDB 等）
- [ ] 正文无接口名、方法名、类名（PolicyAppealApi、create() 等）
- [ ] 正文无表名、字段名（policy_appeal、gmt_create 等）
- [ ] 正文无中间件/协议名（MQ、RPC、Dubbo、HTTP、REST 等）
- [ ] 正文无服务/模块名（policy-appeal-service 等）
- [ ] 技术细节若需保留，已放入 §7.3「内部参考（仅供研发接力）」并标注「待研发确认」
- [ ] 影响面与冲突分析已从技术事实转写为业务表述

---

## 三、一致性

- [ ] 需求描述与 `knowledge/` 现有文档无矛盾（或已标注分析盲区）
- [ ] 影响面与架构文档（INDEX_GUIDE.md）一致
- [ ] 文档内部 G-n / Q-n / C-n / R-n 编号无重复、无断号
- [ ] 各章节交叉引用自洽（目标→范围→影响→方案→风险）

---

## 四、可追溯性

- [ ] 每个 G-n 可追溯到原始业务描述中的具体段落
- [ ] 每个受影响功能可追溯到具体业务能力/协作环节（非系统模块名）
- [ ] 每个 C-n 可追溯到具体规则、协作约定或业务后果层面的契约影响
- [ ] 每个 R-n 可追溯到冲突分析、方案取舍或资源约束

---

## 五、格式规范

- [ ] 文末「## 文档元数据」内 fenced YAML 完整（id、title、version、status、created、updated、author、parent、dependencies、tags）；**文件开头无** `---` YAML frontmatter；元数据仅在文末一处
- [ ] `id` 格式符合 `SOLUTION-{IDEA-ID}`，与文件名一致，`status` 初始为 `draft`
- [ ] 章节结构与 solution-template.md **七章**及模板内 `###` / `####` 小节标题一致，无删除章节
- [ ] 无内容小节已标注「不适用」或「待补充」，未留空
- [ ] 表格格式规范，列宽合理
- [ ] 术语表（§7.1）涵盖本文档专用术语
