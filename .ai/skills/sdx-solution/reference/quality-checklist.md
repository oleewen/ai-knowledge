# 解决方案质量验收清单（sdx-solution）

步骤 5 验收用：Agent 在文档输出与评审时复制本清单并逐项勾选，**所有项通过后才输出文档**。原则与反模式见 [design-principles.md](design-principles.md)；模板结构见 [../assets/solution-template.md](../assets/solution-template.md)。

---

## 一、完整性

- [ ] **§1 业务背景**：需求来源、现状、问题/痛点均已描述
- [ ] **§2 业务目标**：每个 G-n 有度量指标和目标值，无「提升体验」等不可度量表述
- [ ] **§3.1 核心场景**：覆盖主要业务场景，用「谁做什么」描述
- [ ] **§3.2 用户角色**：列出所有涉及角色及关注点
- [ ] **§3.3 约束条件**：业务/资源约束完整
- [ ] **§3.5 待澄清项**：歧义/缺失信息已标注为 Q-n，无自行假设
- [ ] **§4 影响面**：覆盖功能、数据、接口（对外承诺）、下游四个维度；传播路径清晰
- [ ] **§5 冲突分析**：业务与系统冲突完整覆盖（或已标注 `--skip-conflict` 且场景合法）；每项含化解成本
- [ ] **§6.1 解决思路**：整体策略明确
- [ ] **§6.2 关键决策**：每项有推荐方案、备选方案与选定理由
- [ ] **§6.4 范围界定**：In Scope / Out of Scope 清晰
- [ ] **§7 可行性**：技术可行性（业务语言）、资源评估、风险 R-n 完整，每个 R-n 有应对策略
- [ ] **§8 MVP 拆分**：每个 MVP 有独立业务价值，可独立演示给业务方

---

## 二、业务语言

- [ ] 正文无具体技术栈名称（Spring Boot、Kafka、Redis、TiDB 等）
- [ ] 正文无接口名、方法名、类名（PolicyAppealApi、create() 等）
- [ ] 正文无表名、字段名（policy_appeal、gmt_create 等）
- [ ] 正文无中间件/协议名（MQ、RPC、Dubbo、HTTP、REST 等）
- [ ] 正文无服务/模块名（policy-appeal-service 等）
- [ ] 技术细节若需保留，已放入 §9.3「内部参考」并标注「待研发确认」
- [ ] 影响面与冲突分析已从技术事实转写为业务表述

---

## 三、一致性

- [ ] 需求描述与 `knowledge/` 现有文档无矛盾（或已标注分析盲区）
- [ ] 影响面与架构文档（INDEX_GUIDE.md）一致
- [ ] 文档内部 G-n / C-n / R-n 编号无重复、无断号
- [ ] 各章节交叉引用自洽（目标→需求→方案→风险）

---

## 四、可追溯性

- [ ] 每个 G-n 可追溯到原始业务描述中的具体段落
- [ ] 每个受影响功能可追溯到具体业务能力/协作环节（非系统模块名）
- [ ] 每个 C-n / C-Tn 可追溯到具体规则或协作约定
- [ ] 每个 R-n 可追溯到冲突分析或可行性评估

---

## 五、格式规范

- [ ] frontmatter 完整（id、title、version、status、created、updated）
- [ ] `id` 格式符合 `SOLUTION-{YYMMDD}-{IDEA}`，`status` 初始为 `draft`
- [ ] 章节结构与 solution-template.md 九章一致，无删除章节
- [ ] 无内容章节已标注「不适用」或「待补充」，未留空
- [ ] 表格格式规范，列宽合理
- [ ] 术语表（§9.1）涵盖本文档专用术语
