# 解决方案质量验收清单（sdx-solution）

步骤 5 验收用：Agent 在文档输出与评审时**打开本文件**并对照模板 **§7.4** 逐项落实。**完整通过标准**以 [../assets/solution-template.md](../assets/solution-template.md) **§7.4 质量自查表** 正文为准；下表为快速执行摘要。**定稿写入 `SOLUTION-*.md` 时**：对已达标的条目，须在正文 **§7.4** 将 `- [ ]` 改为 `- [x]`；未达标项保持 `- [ ]` 直至修复，不得虚假勾选。

原则与反模式见 [design-principles.md](design-principles.md)。

## 一、完整性（章节与表格）

- [ ] **§1.1–§1.4**：现状、问题、业务目标（G-n）、业务价值均已覆盖
- [ ] **§2.1–§2.4**：核心场景、角色、范围边界（含 In/Out/成功标准）、关键约束（含交付节点）
- [ ] **§3.1–§3.3**：叙述性影响面、§3.2 表、§3.3 传播路径；覆盖功能、数据、对外承诺、下游四维度
- [ ] **§3.4**：冲突表完整（或 `--skip-conflict` 合法）；C-n 含化解成本；协作/契约类以业务后果表述
- [ ] **§4.1–§4.3**：解决思路、方案对比（或多方案说明）、关键决策
- [ ] **§5.1–§5.2**：R-n、Q-n 表列完整
- [ ] **§6.1–§6.2**：MVP、里程碑
- [ ] **§7.1–§7.4**：术语、参考、内部参考（按需）、**质量自查表已勾选**

## 二、业务语言

- [ ] 正文无具体技术栈名称（Spring Boot、Kafka、Redis、TiDB 等）
- [ ] 正文无接口名、方法名、类名
- [ ] 正文无表名、字段名
- [ ] 正文无中间件/协议名、服务/模块名
- [ ] 技术细节若需保留，已放入 §7.3 并标注「待研发确认」
- [ ] 影响面与冲突分析已从技术事实转写为业务表述

---

## 三、一致性

- [ ] 需求描述与 `knowledge/` 无矛盾（或已标注分析盲区）
- [ ] 影响面与 INDEX_GUIDE 等领域索引一致或已说明偏差
- [ ] 文档内部 G-n / Q-n / C-n / R-n 编号无重复、无断号（或已声明原因）
- [ ] 各章节交叉引用自洽（目标→范围→影响→方案→风险）

---

## 四、可追溯性

- [ ] 每个 G-n 可追溯到原始业务描述
- [ ] 每个受影响项可追溯到业务能力/协作环节（非系统模块名）
- [ ] 每个 C-n 可追溯到规则、协作约定或业务后果层面的契约影响
- [ ] 每个 R-n 可追溯到冲突分析、方案取舍或资源约束

---

## 五、格式规范

- [ ] 文末「## 文档元数据」内 fenced YAML 完整（id、title、version、status、created、updated、author、parent、dependencies、tags）
- [ ] **文件开头无** `---` YAML frontmatter；元数据仅在文末
- [ ] `id` 格式符合 `SOLUTION-{IDEA-ID}`，与文件名一致
- [ ] 章节结构与 solution-template.md **七章**及模板内小节标题一致
- [ ] 无内容小节已标注「不适用」或「待补充」
- [ ] 术语表（§7.1）涵盖本文档专用术语
