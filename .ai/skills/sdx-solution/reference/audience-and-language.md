# 受众与文档语言（sdx-solution）

解决方案文档面向**业务方与产品**，用于立项前对齐、范围与优先级共识，而非研发实现说明。算法与章节落位见 [workflow-spec.md](workflow-spec.md)；编号与反模式见 [design-principles.md](design-principles.md)。

---

## 受众定位

| 维度 | 要求 |
|------|------|
| 主要读者 | 业务方、产品经理；用于立项前对齐、范围与优先级共识 |
| 次要读者 | 项目负责人、架构师（仅用于评估可行性，不作为实现依据） |
| 不适用读者 | 开发工程师（开发依据为下游 PRD + ADD，不是本文档） |

---

## 语言规则

### 正文必须使用业务语言

用「谁能做什么」「流程如何变」「对外承诺是什么」代替系统内部构造。

| 业务语言（✅ 写入正文） | 技术语言（❌ 禁止写入正文） |
|----------------------|--------------------------|
| 快递网点操作员可以发起申诉 | 调用 PolicyAppealApi.create 接口 |
| 审核结果在 2 小时内同步给申请方 | 通过 MQ 消息异步推送审核结果 |
| 申诉单提交后 24 小时未操作自动作废 | 定时任务扫描 policy_appeal 表状态字段 |
| 财经中心可批量导入申诉明细 | 使用 Excel 解析 + 分布式锁防重 |
| 申诉流程分三个阶段：提交→审核→结算 | 状态机：DRAFT→SUBMITTED→APPROVED→SETTLED |

### 禁止写入正文的内容

- 具体技术栈（Spring Boot、Kafka、Redis、TiDB 等）
- 接口名、方法名、类名（PolicyAppealApi、create()）
- 表名、字段名（policy_appeal、gmt_create）
- 中间件名称（MQ、RPC、Dubbo、HTTP）
- 服务/模块名（policy-appeal-service、policy-appeal-infrastructure）
- 协议名（REST、gRPC）
- 框架/工具名（MyBatis、Maven、Docker）

### 例外：附录「内部参考」

若确需保留可追溯线索，放入附录 §9.3「内部参考（仅供研发接力）」，并标注「待研发确认」。

## Agent 内部分析 vs 文档输出

Agent 为准确评估影响与冲突，**可以**按需查阅 `knowledge/`、AGENTS.md 等工程材料。

但写入 `SOLUTION-{ID}.md` 时，**必须**将技术事实转写为业务表述：

```
内部分析（可用）：
  MS-001 PolicyAppeal 服务的 create 接口需新增 waybillNo 字段校验

写入文档（转写后）：
  申诉单创建流程新增运单号有效性校验，无效运单号将被拒绝
```

---

## 常见转写模式

| 技术事实 | 业务表述转写 |
|----------|------------|
| 新增数据库表字段 | 申诉单新增「XX信息」记录 |
| 接口新增参数 | 申诉创建时需提供「XX」 |
| 状态机新增状态 | 申诉流程新增「XX」阶段 |
| 引入分布式锁 | 防止同一申诉单被重复提交 |
| 数据迁移脚本 | 历史申诉数据需完成格式升级 |
| 新增 MQ 消费者 | 审核结果将自动触发下游结算流程 |
| 接口版本升级 | 现有对接方需在 XX 日期前完成适配 |

## 自查问题

在输出文档前，逐段自问：

1. 这段话，业务方能看懂吗？
2. 这段话，是否暴露了不必要的系统内部构造？
3. 这段话，是否用「谁做什么」而非「系统调用什么」来表述？
4. 如果有技术词，是否已放入附录或标注「待研发确认」？
