# 文档升级（docs-upgrade）

面向仓库文本的**定向编辑**；是否在多文件间做**链式同步**取决于本次是「新增」还是「修改」（见下节）。

## 新增 vs 修改（范围边界）

| 模式 | 落盘范围 | 链式更新 |
|------|----------|----------|
| **新增** | **仅**用户明确指定的文件（可多个路径，但不得超出该集合） | **不做**：不自动改入站/出站关联文档 |
| **修改** | 用户指定的**任意**文件（一个或多个路径，跨目录亦可） | **可做**：对每个已改文件分别建关联集，更新引用/被引用处同类表述（见「主流程」） |

判定：

- 用户意图为「新建文件 / 追加章节 / 只往某文件里加内容」且未要求同步别处 → 按 **新增** 规则，只动指定文件。
- 用户意图为「改表述、替换术语、修订结论、删改段落」或明确说「同步相关文档」→ 按 **修改** 规则，允许链式。
- 模糊时先问一句：「本次是否只需改指定文件，还是要一并更新引用链上的文档？」

## 触发即执行

一旦本技能适用，**直接改文件**完成用户目标；不要只给操作说明。若范围过大（例如全库术语替换），先与用户确认边界或分批。

## 输入解析

| 用户表达 | 处理 |
|----------|------|
| 明确路径（如 `docs/solutions/X.md`） | 以该文件为主文档；**新增**则只改这些路径，**修改**则可继续链式 |
| 「替换」简写 | 见下节「替换简写」 |
| 自然语言增改要点 | 写入主文档对应章节；**仅在为修改模式时**再跑关联链 |
| 未给路径但给主题 | 用 `rg`/搜索定位最相关文件，**若多候选则列出选项**再编辑 |

## 替换简写

以下等价：**将左侧替换为右侧**（trim 首尾空白后匹配）。

| 形式 | 含义 |
|------|------|
| `a - b` | `a` → `b` |
| `a > b` | `a` → `b` |
| `a 2 b` | `a` → `b`（数字 2 表示 to） |

规则：

- 若 `a` 或 `b` 含 `|`、换行、或边界不清，改用引号：例如 `"旧术语" - "新术语"`。
- 执行替换前确认 `a` 在目标段落中唯一或用户接受多处同时替换；否则先列出命中次数与示例行。

## 主流程（建议顺序）

### 新增模式

1. **读指定文件**（若无则创建）：确认结构、插入位置。
2. **仅在本集合内落盘**：完成新增段落或新文件内容。
3. **结束**：不执行建关联、不入站/出站检索（除非用户另行要求改为修改模式）。

### 修改模式

1. **读待改文件**：可为多个路径；确认结构、标题层级、既有术语与链接。
2. **落主修改**：在每个指定文件上按用户意图完成改写/替换。
3. **建关联集合**（对每个本轮已修改的文件分别做）：
   - **出站**：从该文件提取 Markdown 链接、显式路径、`@` 文件引用。
   - **入站**：在仓库内检索引用该文件路径或文件名的位置（见 [references/related-doc-discovery.md](references/related-doc-discovery.md)）。
4. **链式更新**：对关联文档中与已改内容**同一概念、重复表述、需同步的句子或术语**做一致化修改；避免改与本次变更无关的独立叙述。
5. **回链校验**：若关联文档也链回主文档，确认链接文字与锚点仍有效。
6. **不确定项门禁**：任何无法在仓库文档或用户指令中核实的内容 → **停止并输出选项**（见下节）。

## 不确定项与用户决策

在下列情况**不得自行编造**；应使用编号选项（可含推荐项与理由）：

- 业务规则、数字指标、日期、责任人、政策结论。
- 「是否同步删除过时段落」「是否保留兼容说明」等结构性选择。
- 多文件同名冲突、多段相似但语义可能不同的文本。

用户选择后再继续编辑。

## 与相邻技能的关系

| 场景 | 建议 |
|------|------|
| 需要全库变更索引、审计时间线 | 使用 `docs-change`，本技能不替代 |
| 需要重建 INDEX_GUIDE 九章索引 | 使用 `docs-indexing` |
| 仅改代码不改文档 | 不用本技能 |

## 验证

- 修改后快速检查：链接可解析、代码块与表格未破坏、术语与主文档关键句一致。
- 用户要求时运行仓库既有校验（如 README 中的检查命令）。
