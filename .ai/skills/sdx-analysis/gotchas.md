# sdx-analysis 常见陷阱

---

## 输入与前置

**无解决方案文档直接开始分析**：解决方案文档（`system/solutions/SOLUTION-{ID}.md`）是硬输入；不存在时终止并提示先执行 `sdx-solution`。FR-n 无法追溯到 G-n 时，可追溯性链条从根部断裂。

**解决方案文档结构不完整时静默继续**：发出警告并列出缺失章节，基于已有内容继续并在文档中显式标注「基于不完整解决方案，存在分析盲区」。

**knowledge 目录缺失时不标注**：在 §1.3 研究发现中显式标注「缺少知识库基线，以下分析仅基于解决方案文档」，不凭印象假设现有系统状态。

---

## 步骤 1：深度研究

**通读全仓知识库**：从解决方案文档的影响面评估出发，按图索骥——只打开与本次业务目标直接相关的知识库视角文件，不通读 `knowledge/**`。

**忽略 `--depth` 参数**：`quick` 仅研究领域边界与核心规则；`standard` 完整四维度；`deep` 额外增加行业对标与性能基线调研。无论哪种深度，写入文档时均须转为业务表述。

**研究发现未记录知识库盲区**：解决方案中提到的概念在知识库中找不到对应实体时，在 §1.3 中显式标记「研究盲区」，说明哪些概念缺失，避免下游重复踩坑。

---

## 步骤 2：需求细化

**功能需求缺少优先级**：每个 FR-n 必须标注优先级（P0–P3）；P0 为必须交付，P3 为可选；优先级缺失时暂停并请求用户确认，否则步骤 3 MVP 拆分无依据。

**非功能需求描述模糊**：性能需求须量化（如「P99 响应时间 < 200ms」），不写「响应速度快」「高可用」等无法度量的描述；无法量化时标注「待澄清」。

**业务规则嵌入功能需求描述**：业务规则独立编号为 BR-n，在 FR-n 中通过「关联 BR-n」字段引用，不嵌入 FR-n 描述字段，否则规则无法被多个 FR-n 复用引用。

**吞没歧义自行补全**：歧义项标注为「待澄清」，暂停并请求用户澄清后再继续，不自行假设后继续细化。

---

## 步骤 3：MVP 拆分

**P0 需求出现在后序 MVP**：P0 需求必须在首个 MVP 中交付；技术基础设施随首个消费它的 MVP 一并交付。首个 MVP 无法独立交付业务价值时违背 MVP 拆分核心原则。

**MVP 间出现循环依赖**：拆分后绘制 MVP 间有向无环图验证；发现循环依赖时终止拆分，输出依赖图请求用户确认调整。

**按功能维度机械拆分而非业务价值**：每个 MVP 必须能独立演示给业务方；自问「这个 MVP 能解决什么业务问题」，答不上来则重拆。

**MVP 缺少验收标准**：每个 MVP 必须有明确的可度量验收条件，否则交付时无法判断是否完成。

---

## 步骤 4：依赖与风险

**风险无编号、无应对策略**：每个风险编号为 R-n，标注可能性（高/中/低）与影响（高/中/低）；高风险必须有具体应对策略与监控指标，不写「存在性能风险，需要关注」等自由文本。

**外部依赖未纳入风险评估**：依赖分析须覆盖四类：功能依赖、数据依赖、接口依赖、外部依赖；外部依赖须评估 SLA 与降级方案。

---

## 步骤 5：文档输出

**跳过或重排模板章节**：章节结构与 `analysis-template.md` 完全一致；无内容的章节保留标题并标注「不适用」或「待补充」，不删除。下游 sdx-prd 按固定章节引用，章节缺失或错位导致引用失效。

**文档元数据位置错误或字段缺失**：勿在文件开头写 YAML frontmatter；仅在文末「## 文档元数据」标题下的 fenced `yaml` 中填写字段；须包含 `id`、`title`、`version`、`status`、`created`、`updated`、`parent`，初始 `status` 为 `draft`，`parent` 填写上游解决方案编号。

**在需求分析中写入 PRD 级内容**：需求分析只描述「做什么」和「验收标准」，不描述「怎么做」；用户故事（As a... I want...）和技术实现细节（如「使用 Redis 缓存」）留给下游 Skill。

**跳过质量门禁自查**：步骤 5 必须逐项检查 [reference/quality-checklist.md](reference/quality-checklist.md)，所有检查项通过后才输出文档。

## 快速自查清单

- [ ] 解决方案文档已落盘且可用，`parent` 字段已填写
- [ ] 研究阶段按 `--depth` 参数调整粒度，未通读全仓
- [ ] 知识库盲区已在 §1.3 显式标注
- [ ] 每个 FR-n 有编号、描述、优先级（P0–P3）、验收标准
- [ ] 非功能需求已量化，无模糊描述
- [ ] 业务规则独立编号为 BR-n，未嵌入 FR-n 描述
- [ ] 歧义项已标注「待澄清」，未自行假设
- [ ] P0 需求全部在首个 MVP 中交付
- [ ] MVP 间无循环依赖（有向无环图已验证）
- [ ] 每个 MVP 有可度量验收标准
- [ ] 每个风险编号为 R-n，高风险有应对策略
- [ ] 章节结构与 `analysis-template.md` 完全一致，无删除章节
- [ ] 文末元数据 YAML 字段完整（含 `parent`）；文件开头无 `---` YAML
- [ ] 文档中无 PRD 级用户故事或技术实现细节
- [ ] quality-checklist.md 已逐项勾选
