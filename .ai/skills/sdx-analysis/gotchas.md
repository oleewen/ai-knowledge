# sdx-analysis 常见陷阱（Gotchas）

本文记录执行 `sdx-analysis` Skill 时高频踩坑点，供 Agent 与人类开发者参考。

---

## 1. 输入与前置条件

### 1.1 无解决方案文档直接开始分析
**陷阱**：用户直接描述业务需求，Agent 跳过 `sdx-solution` 直接输出需求分析文档。  
**后果**：缺少结构化的业务目标（G-n）作为需求追溯锚点，FR-n 无法追溯，可追溯性链条从根部断裂。  
**正确做法**：解决方案文档（`docs/solutions/SOLUTION-{ID}.md`）是硬输入；不存在时终止并提示先执行 `sdx-solution`。

### 1.2 解决方案文档结构不完整时静默继续
**陷阱**：解决方案文档缺少业务目标章节或影响面评估，Agent 未发出警告直接推进。  
**后果**：基于残缺输入生成的需求分析同样残缺，且用户无感知。  
**正确做法**：发出警告并列出缺失章节，基于已有内容继续并在文档中显式标注「基于不完整解决方案，存在分析盲区」。

### 1.3 knowledge 目录缺失时不标注
**陷阱**：`knowledge/` 目录不存在，Agent 仅基于解决方案文档完成分析，但未在文档中说明。  
**后果**：读者误以为分析已参照知识库基线，实际上业务规则、领域边界均未经知识库验证。  
**正确做法**：在研究发现（§1.3）中显式标注「缺少知识库基线，以下分析仅基于解决方案文档」。

---

## 2. 步骤 1：深度研究

### 2.1 通读全仓知识库
**陷阱**：步骤 1 打开 `knowledge/**` 下所有文件进行「全面研究」。  
**后果**：上下文爆炸，耗时极长，且大量内容与本次解决方案无关。  
**正确做法**：从解决方案文档的影响面评估出发，按图索骥——只打开与本次业务目标直接相关的知识库视角文件。

### 2.2 忽略 `--depth` 参数调整研究粒度
**陷阱**：无论用户传入 `quick` 还是 `deep`，始终执行完整四维度研究。  
**后果**：`quick` 场景下耗时过长；`deep` 场景下缺少行业对标与技术可行性预研。  
**正确做法**：`quick` 仅研究领域边界与核心规则；`standard` 完整四维度；`deep` 额外增加行业对标与性能基线调研。

### 2.3 研究发现未记录知识库盲区
**陷阱**：解决方案中提到的概念在知识库中找不到对应实体，Agent 跳过不记录。  
**后果**：下游 `sdx-prd` 和 `sdx-design` 遇到同样盲区时无预警，重复踩坑。  
**正确做法**：在 §1.3 研究发现中显式标记「研究盲区」，说明哪些概念在知识库中缺失。

---

## 3. 步骤 2：需求细化

### 3.1 功能需求缺少优先级
**陷阱**：FR-n 只有描述，未标注 P0–P3 优先级。  
**后果**：步骤 3 MVP 拆分时无法判断哪些需求必须进入首个 MVP，拆分结果不可靠。  
**正确做法**：每个 FR-n 必须标注优先级；P0 为必须交付，P3 为可选；优先级缺失时暂停并请求用户确认。

### 3.2 非功能需求描述模糊
**陷阱**：性能需求写「响应速度快」，可用性写「高可用」，无具体指标。  
**后果**：验收标准无法度量，测试设计无依据，上线后扯皮。  
**正确做法**：非功能需求必须量化，如「P99 响应时间 < 200ms」「SLA ≥ 99.9%」；无法量化时标注「待澄清」。

### 3.3 业务规则嵌入功能需求描述
**陷阱**：将业务规则直接写在 FR-n 的描述字段中，不单独提取为 BR-n。  
**后果**：业务规则无独立编号，无法被多个 FR-n 复用引用，也无法单独追溯。  
**正确做法**：业务规则独立编号为 BR-n，在 FR-n 中通过 `关联 BR-n` 字段引用。

### 3.4 吞没歧义自行补全
**陷阱**：解决方案文档中某需求描述模糊，Agent 自行假设后继续细化。  
**后果**：需求分析基于错误假设，下游 PRD 和设计方案全部偏离。  
**正确做法**：歧义项标注为「待澄清」，在文档中显式记录疑问点，暂停并请求用户澄清后再继续。

---

## 4. 步骤 3：MVP 拆分

### 4.1 P0 需求出现在后序 MVP
**陷阱**：MVP-1 只包含基础框架，P0 核心功能被推到 MVP-2 或 MVP-3。  
**后果**：首个 MVP 无法独立交付业务价值，违背 MVP 拆分的核心原则。  
**正确做法**：P0 需求必须在首个 MVP 中交付；技术基础设施随首个消费它的 MVP 一并交付。

### 4.2 MVP 间出现循环依赖
**陷阱**：MVP-2 依赖 MVP-3 的某个接口，MVP-3 又依赖 MVP-2 的数据。  
**后果**：无法确定交付顺序，两个 MVP 实际上必须同时交付，失去拆分意义。  
**正确做法**：拆分后绘制 MVP 间有向无环图验证；发现循环依赖时终止拆分，输出依赖图请求用户确认调整。

### 4.3 按功能维度机械拆分而非业务价值
**陷阱**：MVP-1 做「所有查询接口」，MVP-2 做「所有写入接口」，按技术类型而非业务场景拆分。  
**后果**：每个 MVP 无法独立演示给业务方，交付价值不可见。  
**正确做法**：每个 MVP 必须能独立演示给业务方；拆分后自问「这个 MVP 能解决什么业务问题」，答不上来则重拆。

### 4.4 MVP 缺少验收标准
**陷阱**：MVP 只有功能清单，没有可度量的验收条件。  
**后果**：交付时无法判断 MVP 是否完成，验收扯皮。  
**正确做法**：每个 MVP 必须有明确的验收标准，格式与 FR-n 验收标准一致，可度量。

---

## 5. 步骤 4：依赖与风险

### 5.1 风险无编号、无应对策略
**陷阱**：风险评估写成自由文本段落，如「存在性能风险，需要关注」。  
**后果**：风险无法追溯，无监控指标，高风险项无应对策略，上线后被动应对。  
**正确做法**：每个风险编号为 R-n，标注可能性（高/中/低）与影响（高/中/低）；高风险必须有具体应对策略与监控指标。

### 5.2 外部依赖未纳入风险评估
**陷阱**：分析了 MVP 间的内部依赖，但遗漏了对第三方服务、基础设施的外部依赖。  
**后果**：外部依赖阻塞时无预案，进度风险被低估。  
**正确做法**：依赖分析必须覆盖四类：功能依赖、数据依赖、接口依赖、外部依赖；外部依赖须评估 SLA 与降级方案。

---

## 6. 步骤 5：文档输出

### 6.1 跳过或重排模板章节
**陷阱**：认为某章节「不适用」直接删除，或调整章节顺序以「更符合逻辑」。  
**后果**：下游 `sdx-prd` 按固定章节引用需求分析文档，章节缺失或错位导致引用失效。  
**正确做法**：章节结构与 `analysis-template.md` 完全一致；无内容的章节保留标题并标注「不适用」或「待补充」，不得删除。

### 6.2 frontmatter 字段缺失
**陷阱**：输出文档缺少 `parent`（关联解决方案编号）或 `status` 字段。  
**后果**：文档与上游解决方案的追溯链断裂；`status` 缺失导致文档管理混乱。  
**正确做法**：frontmatter 必须包含 `id`、`title`、`version`、`status`、`created`、`updated`、`parent` 七个字段，初始 `status` 为 `draft`。

### 6.3 在需求分析中写入 PRD 级内容
**陷阱**：在 FR-n 详述中写出用户故事（As a... I want... So that...）或技术实现细节（如「使用 Redis 缓存」）。  
**后果**：范围蔓延，需求分析与 PRD 职责混淆，下游 `sdx-prd` 无法在此基础上增量工作。  
**正确做法**：需求分析只描述「做什么」和「验收标准」，不描述「怎么做」；用户故事和技术方案留给下游 Skill。

### 6.4 跳过质量门禁自查
**陷阱**：文档写完后直接输出，未逐项检查 `quality-gate-checklist.md`。  
**后果**：编号断号、章节缺失、追溯链断裂等问题流入下游，被 `sdx-prd` 或评审发现时返工成本高。  
**正确做法**：步骤 5 必须执行质量门禁自查，所有检查项通过后才输出文档。

---

## 快速检查清单

执行完毕后，对照以下项目快速自查：

- [ ] 解决方案文档已落盘且可用，`parent` 字段已填写
- [ ] 研究阶段按 `--depth` 参数调整粒度，未通读全仓
- [ ] 知识库盲区已在 §1.3 显式标注
- [ ] 每个 FR-n 有编号、描述、优先级（P0–P3）、验收标准
- [ ] 非功能需求已量化，无模糊描述
- [ ] 业务规则独立编号为 BR-n，未嵌入 FR-n 描述
- [ ] 歧义项已标注「待澄清」，未自行假设
- [ ] P0 需求全部在首个 MVP 中交付
- [ ] MVP 间无循环依赖（有向无环图已验证）
- [ ] 每个 MVP 有可度量验收标准
- [ ] 每个风险编号为 R-n，高风险有应对策略
- [ ] 章节结构与 `analysis-template.md` 完全一致，无删除章节
- [ ] frontmatter 七个字段完整
- [ ] 文档中无 PRD 级用户故事或技术实现细节
- [ ] 质量门禁清单已逐项勾选
