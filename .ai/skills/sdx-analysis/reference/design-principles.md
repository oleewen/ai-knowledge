# 设计原则与反模式

sdx-analysis 技能的完整设计约束。[SKILL.md](../SKILL.md) 中的「核心约束」为精简版，本文件为完整规范。验收勾选项见 [quality-checklist.md](quality-checklist.md)。

---

## 设计原则

### 1. 模板驱动

输出严格遵循 [../assets/analysis-template.md](../assets/analysis-template.md) 的八章结构（需求概述→功能需求→非功能需求→业务规则→数据需求→MVP 拆分方案→依赖与风险→附录）。章节顺序与编号不可调整；无内容的章节保留标题并标注「不适用」或「待补充」。

### 2. 证据优先

需求分析须引用解决方案文档与 `knowledge/` 事实，禁止凭空臆测。引用格式遵循项目规范：

| 证据类型 | 格式 | 示例 |
|----------|------|------|
| 解决方案目标 | `G-{N}` | `G-1 提升申诉处理效率` |
| 知识库实体 | `{视角}-{ID}` | `BC-001 FeeAppealContext` |
| 文档章节 | `{文件} §{章节}` | `INDEX_GUIDE.md §3.2` |
| 代码位置 | `{类}:{方法}` | `BillingAppealService:create` |

### 3. 按需加载

仅在本轮任务需要时打开文件：

- 步骤 1 读解决方案文档 + 对应知识库视角
- 步骤 2 按需求类型打开业务/技术/数据知识库
- 步骤 3 综合已读内容拆分 MVP
- 步骤 4 打开相关规约与架构文档评估依赖

禁止为「完整性」通读 `knowledge/**` 或全仓源码。

### 4. 歧义标注

对解决方案文档或知识库中的歧义、矛盾、信息缺失，标注为待澄清项，在文档中显式记录。禁止自行假设缺失信息后继续，应暂停并请求用户澄清。

### 5. 范围边界

本技能仅负责输出需求分析文档。不产出 PRD、ADD、测试设计或代码——那些是下游 sdx-prd / sdx-design / sdx-test 的职责。

### 6. 可追溯性

- 每个功能需求（FR-n）可追溯到解决方案中的业务目标（G-n）
- 每个业务规则（BR-n）可追溯到功能需求（FR-n）
- 每个 MVP 范围可追溯到功能需求清单
- 每个风险（R-n）可追溯到依赖分析或影响面评估

### 7. ID 编号规范

需求分析文档使用以下 ID 编号体系：

| 前缀 | 用途 | 示例 |
|------|------|------|
| `ANALYSIS-{ID}` | 文档编号 | `ANALYSIS-260327-申诉处理提效` |
| `FR-{NNN}` | 功能需求编号 | `FR-001`、`FR-002` |
| `BR-{NNN}` | 业务规则编号 | `BR-001`、`BR-002` |
| `R-{N}` | 风险编号 | `R-1`、`R-2` |
| `MVP-{N}` | MVP 阶段编号 | `MVP-1`、`MVP-2` |

## 反模式清单（禁止）

| 反模式 | 说明 |
|--------|------|
| 臆测需求 | 未引用解决方案或 knowledge 即自行补充功能需求 |
| 吞没歧义 | 将不明确需求自行补全后继续，不标注待澄清 |
| 范围蔓延 | 在需求分析中直接写出 PRD 级别的用户故事或技术设计细节 |
| 模板跳章 | 跳过或重排 analysis-template.md 的章节结构 |
| 通读全库 | 一次性读取全仓知识库而不考虑本轮任务实际需要 |
| 无编号引用 | 需求/规则/风险无统一编号，导致不可追溯 |
| MVP 硬拆 | 不考虑业务价值独立性，机械按功能维度拆分 MVP |
| 循环依赖 | MVP 间出现双向依赖 |
| 空章提交 | 章节标题下无任何内容且未标注「不适用」或「待补充」 |
| 优先级缺失 | 功能需求未标注 P0–P3 优先级 |

## 最佳实践

### 深度研究

- 从解决方案文档的影响面评估出发，按图索骥查阅知识库
- 识别解决方案中提到但知识库中缺失的概念，标记为研究盲区
- 对数字指标（时间、量级、频率）与解决方案文档交叉验证

### 需求细化

- 从解决方案的核心场景出发，逐层分解为可实现的功能需求
- 非功能需求尽量量化（如 P99 < 200ms），避免模糊描述（如「性能良好」）
- 业务规则独立编号，避免嵌入功能需求描述中

### MVP 拆分

- 优先考虑能快速验证核心假设的最小功能集
- 技术基础设施（如新表、新接口）跟随首个消费它的 MVP
- 拆分后回顾：每个 MVP 能否独立演示给业务方

### 风险评估

- 从依赖分析中识别阻塞链路上的关键节点
- 技术风险优先通过 POC/Spike 验证
- 进度风险关注外部依赖与跨团队协作

## 错误处理

| 错误场景 | 处理方式 |
|----------|----------|
| 解决方案文档不存在 | 终止，提示先执行 sdx-solution |
| 解决方案文档结构不完整 | 发出警告，列出缺失章节，基于已有内容继续并标注风险 |
| knowledge 目录缺失 | 发出警告，仅基于解决方案文档完成分析，标注缺少知识库基线 |
| 模板文件不存在 | 终止，提示创建 `.ai/skills/sdx-analysis/assets/analysis-template.md` |
| MVP 拆分出现循环依赖 | 终止拆分，输出依赖图，请求用户确认调整方案 |
| 输出目录不存在 | 自动创建 `system/analysis/` 目录 |
