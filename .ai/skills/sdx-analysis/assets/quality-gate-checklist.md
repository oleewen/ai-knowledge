# 需求分析质量门禁清单

步骤 5 验收模板。Agent 在执行文档输出与评审时复制本清单并逐项勾选。

## 完整性

- [ ] 需求概述（§1）：需求背景引用解决方案文档，目标明确
- [ ] 研究发现（§1.3）：边界、规则、协作与风险已记录；**无不当技术细节散落于正文**
- [ ] 功能需求清单（§2.1）：每个 FR-n 有编号、描述、优先级、所属 MVP、关联 G-n
- [ ] 功能需求详述（§2.2）：每个 FR-n 有输入、处理逻辑、输出、业务规则、异常与边界、验收标准（业务语义）
- [ ] 非功能需求（§3）：体验与性能期望、可用性与连续性、安全与合规、运维与问题定位诉求、兼容与升级节奏各节完整（以业务可读表述为主）
- [ ] 业务规则（§4）：每个 BR-n 有触发条件、执行逻辑、优先级、关联 FR-n
- [ ] 数据需求（§5）：业务对象/信息变更清晰，一致性与时效期望为业务话术
- [ ] MVP 拆分（§6）：每个 MVP 有目标、范围、验收标准、预估工期、依赖（协作与流程表述）
- [ ] 依赖与风险（§7）：依赖完整，风险 R-n 有可能性/影响/应对策略（优先业务与协作影响）

## 受众与语言

- [ ] 正文符合 [.ai/skills/sdx-solution/reference/audience-language-spec.md](../../sdx-solution/reference/audience-language-spec.md)：产品/需求分析师与业务方可独立理解主流程
- [ ] 接口名、表名、中间件、服务/模块名**未出现在正文**（或仅集中在模板 §8.4 并标注待研发确认）

## 一致性

- [ ] 功能需求 FR-n 与解决方案 G-n 一一对应，无遗漏
- [ ] 业务规则与 `knowledge/business/` 现有规则无矛盾
- [ ] MVP 拆分与需求优先级一致（P0 需求在前序 MVP）
- [ ] 文档内部 FR-n / BR-n / R-n / MVP-n 编号连续无断号
- [ ] 各章节交叉引用自洽（需求→规则→MVP→风险）

## 可行性

- [ ] MVP 拆分粒度合理，每个可独立交付与演示
- [ ] 依赖关系无循环，MVP 间单向依赖
- [ ] 工作量估算有参照基准
- [ ] 高风险项（R-n）有具体应对策略与可跟进行动

## 可追溯性

- [ ] 每个 FR-n 可追溯到解决方案 G-n
- [ ] 每个 BR-n 可追溯到 FR-n
- [ ] 每个 MVP-n 可追溯到 FR-n 列表
- [ ] 每个 R-n 可追溯到依赖分析或影响面评估

## 格式规范

- [ ] frontmatter 完整（id、title、version、status、created、updated、parent）
- [ ] 章节结构与 analysis-template.md 一致，编号连续
- [ ] 表格格式规范，列宽合理
- [ ] 术语表（§8.1）涵盖本文档专用术语
- [ ] 参考文档（§8.2）列出引用的解决方案/知识库文件
- [ ] 变更历史（§8.3）记录初始版本
- [ ] 模板 §8.5 质量自查表已勾选（若采用模板内嵌自查）
