# sdx-test 常见陷阱（Gotchas）

本文记录执行 `sdx-test` Skill 时高频踩坑点，供 Agent 与人类开发者参考。

---

## 1. 输入与前置条件

### 1.1 无 PRD 直接开始测试设计
**陷阱**：用户描述功能需求，Agent 跳过 `sdx-prd` 直接输出 TDD。  
**后果**：缺少用户故事（US-n）和业务规则（BR-n）作为测试追溯锚点，测试用例无法追溯到业务需求。  
**正确做法**：PRD 文档是硬输入；不存在时终止并提示先执行 `sdx-prd`。

### 1.2 未限定目标 MVP 范围
**陷阱**：未读取 PRD 中的 MVP 边界，将所有 MVP 的用户故事一并纳入测试设计。  
**后果**：TDD 超越 MVP 边界，测试范围虚增，测试资源分配失准，下游执行阶段无法聚焦。  
**正确做法**：严格聚焦 `--mvp` 参数指定的 MVP 范围；PRD 中该 MVP 未包含的 US-n 一律不纳入本次 TDD。

### 1.3 ADD 缺失时不标注，凭印象假设接口
**陷阱**：`ADD-{ID}.md` 不存在，Agent 凭经验猜测接口签名、错误码、幂等性设计，直接编写接口测试用例。  
**后果**：接口测试用例基于幻觉，与实际实现不符，测试执行阶段大量用例失效。  
**正确做法**：ADD 缺失时在文档中显式标注「缺少架构设计文档，接口测试用例基于 PRD 推断，需开发完成后核对」；接口测试用例标注 `待核实`。

---

## 2. 步骤 1：测试策略与范围分析

### 2.1 测试层次分配不合理，全部堆在端到端
**陷阱**：测试策略将所有用例设计为端到端测试，未分配单元测试与集成测试比例。  
**后果**：测试金字塔倒置，执行成本极高，反馈周期长，CI/CD 流水线无法快速验证。  
**正确做法**：测试层次须按金字塔原则分配——单元测试覆盖核心业务逻辑，集成测试覆盖服务间交互，端到端测试覆盖关键业务路径；各层次须标注覆盖率目标。

### 2.2 测试范围未区分新增/变更/回归
**陷阱**：测试范围只列出本次新增功能，未识别因变更而需要回归的已有功能。  
**后果**：回归范围遗漏，已有功能被本次变更破坏但未被测试覆盖，上线后出现回归缺陷。  
**正确做法**：测试范围须明确区分三类：新增功能测试、变更功能测试、回归测试；回归范围须基于 PRD/ADD 的影响面评估确定。

### 2.3 测试风险识别流于形式
**陷阱**：测试风险只写「时间紧张」「需求变更」等通用风险，未识别本次 MVP 特有的技术与业务风险。  
**后果**：高风险区域（如复杂业务规则、外部依赖、并发场景）未获得额外测试资源，上线后高风险区域缺陷集中爆发。  
**正确做法**：测试风险须结合本次 MVP 的具体内容识别，如「BR-n 规则逻辑复杂，边界条件多」「依赖外部支付系统，沙箱环境不稳定」；高风险区域须标注加强测试策略。

---

## 3. 步骤 2：测试用例设计

### 3.1 测试用例只覆盖正常场景
**陷阱**：每个用户故事只设计主路径测试用例，未覆盖备选场景、异常场景、边界条件。  
**后果**：异常路径缺少测试覆盖，上线后边界条件触发的缺陷集中暴露。  
**正确做法**：每个用户故事须设计至少三类用例：正常场景（主路径）、异常场景（输入非法/依赖失败/权限不足）、边界条件（最大值/最小值/空值/并发）。

### 3.2 测试用例无追溯编号
**陷阱**：测试用例只有描述，未标注关联的用户故事编号（US-n）、业务规则编号（BR-n）或 API 编号（API-n）。  
**后果**：TDD 与 PRD/ADD 的追溯链断裂，无法验证测试覆盖完整性，缺陷定位时无法快速关联需求。  
**正确做法**：每个测试用例必须标注追溯来源：功能用例关联 `US-n`，规则用例关联 `BR-n`，接口用例关联 `API-n`。

### 3.3 接口测试用例缺少错误码验证
**陷阱**：接口测试用例只验证成功响应结构，未设计错误码场景的测试用例。  
**后果**：错误码处理逻辑未被测试覆盖，前端/调用方的错误处理分支在生产环境才暴露问题。  
**正确做法**：每个 API 的接口测试用例须覆盖 ADD 中定义的所有错误码；每个错误码对应一个独立测试用例，验证触发条件与响应结构。

### 3.4 业务规则测试用例缺少规则冲突场景
**陷阱**：业务规则测试只验证单条规则的触发，未设计多条规则同时触发时的冲突场景。  
**后果**：规则优先级与冲突处理逻辑未被测试覆盖，多规则并发触发时行为不确定。  
**正确做法**：PRD 中标注了优先级或冲突处理策略的 BR-n，须额外设计规则冲突测试用例，验证优先级执行顺序与冲突处理结果。

### 3.5 性能测试用例缺少基准与目标值
**陷阱**：性能测试用例只写「验证系统性能」，未定义并发量、响应时间目标、数据量基准。  
**后果**：性能测试无法判断是否通过，测试结果无法与 PRD/ADD 中的非功能需求对应。  
**正确做法**：性能测试用例须引用 PRD 中的非功能需求指标（如「P99 响应时间 < 200ms，并发 100 用户」）；无指标时标注「待澄清」，不得自行假设。

---

## 4. 步骤 3：测试数据与环境规划

### 4.1 测试数据规划只写「手工准备」
**陷阱**：测试数据需求只标注「手工准备」，未说明数据结构、数量规模、依赖关系。  
**后果**：测试执行阶段数据准备耗时过长，数据不一致导致用例失败，测试进度延误。  
**正确做法**：测试数据规划须包含：数据类型（基础数据/业务数据/边界数据）、准备方式（脚本生成/手工/生产脱敏）、数量规模、数据间依赖关系（如「需先创建用户再创建订单」）。

### 4.2 外部依赖未纳入环境规划
**陷阱**：测试环境规划只列出本服务的部署要求，未说明外部依赖（第三方服务、消息队列、定时任务）的 Mock 或沙箱策略。  
**后果**：测试执行时外部依赖不可用或行为不可控，集成测试用例大量失败，无法区分是本服务缺陷还是外部依赖问题。  
**正确做法**：环境规划须列出所有外部依赖及其测试策略：可用沙箱的使用沙箱，不可用的使用 Mock，并说明 Mock 的覆盖范围与局限性。

---

## 5. 步骤 4：进出标准与回归策略

### 5.1 进入标准过于宽松
**陷阱**：测试进入标准只写「开发完成」，未定义代码审查、单测覆盖率、环境部署验证等具体条件。  
**后果**：代码质量不达标时测试提前介入，大量基础缺陷消耗测试资源，测试效率低下。  
**正确做法**：进入标准须包含可验证的具体条件：代码审查通过、单测覆盖率达标（引用 ADD 中的覆盖率目标）、冒烟测试通过、测试数据就绪。

### 5.2 退出标准缺少缺陷分级标准
**陷阱**：退出标准只写「用例通过率 ≥ 95%」，未定义 P0/P1/P2 缺陷的关闭要求。  
**后果**：存在未关闭的 P0 缺陷时仍满足通过率要求，导致带严重缺陷上线。  
**正确做法**：退出标准须包含缺陷分级关闭要求：P0（阻塞）缺陷必须全部关闭，P1（严重）缺陷关闭率 ≥ 100%，P2（一般）缺陷关闭率 ≥ 80%；各级缺陷定义须在文档中说明。

### 5.3 回归范围凭经验判断，未基于影响面
**陷阱**：回归测试范围由 Agent 凭经验列出「可能受影响的功能」，未参照 PRD/ADD 的影响面评估。  
**后果**：回归范围遗漏或过度，遗漏时上线后出现回归缺陷，过度时测试资源浪费。  
**正确做法**：回归范围须基于 PRD 中的影响面评估章节确定，每个回归测试项须标注关联的影响点（直接影响/间接影响）。

---

## 6. 步骤 5：文档输出与评审

### 6.1 跳过或重排模板章节
**陷阱**：认为某章节「不适用」直接删除，或调整章节顺序。  
**后果**：TDD 章节结构与 `tdd-template.md` 不一致，测试管理工具导入失败，评审时无法按固定结构审查。  
**正确做法**：章节结构与 `tdd-template.md` 六章结构完全一致；无内容的章节保留标题并标注「不适用」或「待补充」，不得删除。

### 6.2 frontmatter 字段缺失
**陷阱**：输出文档缺少 `parent`（关联 PRD 编号）或 `mvp_phase` 字段。  
**后果**：TDD 与上游 PRD 的追溯链断裂；`mvp_phase` 缺失导致多 MVP 场景下管理混乱。  
**正确做法**：frontmatter 必须包含 `id`、`title`、`version`、`status`、`created`、`updated`、`parent`、`mvp_phase` 八个字段，初始 `status` 为 `draft`。

### 6.3 在 TDD 中编写自动化测试代码
**陷阱**：在测试用例设计章节中写出完整 Java/Python 测试代码或 Gherkin 脚本。  
**后果**：范围蔓延，TDD 与代码实现耦合，代码变更时 TDD 难以同步维护，且超出本 Skill 的产出范围。  
**正确做法**：TDD 只描述测试用例的前置条件、操作步骤、预期结果，不写可执行代码；自动化测试脚本由开发阶段产出。

### 6.4 跳过质量门禁自查
**陷阱**：TDD 写完后直接输出，未逐项检查 `quality-gate-checklist.md`。  
**后果**：追溯链断裂、覆盖遗漏、进出标准不可度量等问题流入测试执行阶段，发现时返工成本高。  
**正确做法**：步骤 5 必须执行质量门禁自查，所有检查项通过后才输出文档。

## 快速检查清单

执行完毕后，对照以下项目快速自查：

- [ ] PRD 文档已落盘且可用，`parent` 字段已填写
- [ ] TDD 严格聚焦目标 MVP 范围，未超越 MVP 边界
- [ ] ADD 缺失时已在文档中显式标注，接口用例标注「待核实」
- [ ] 测试层次按金字塔原则分配，各层次有覆盖率目标
- [ ] 测试范围区分新增/变更/回归三类
- [ ] 高风险区域已识别并标注加强测试策略
- [ ] 每个用户故事覆盖正常/异常/边界三类用例
- [ ] 每个测试用例标注追溯来源（US-n / BR-n / API-n）
- [ ] 接口测试用例覆盖 ADD 中定义的所有错误码
- [ ] 多规则冲突场景已设计测试用例
- [ ] 性能测试用例引用 PRD 非功能需求指标，无自行假设
- [ ] 测试数据规划含准备方式、数量规模、数据间依赖关系
- [ ] 外部依赖已列出 Mock 或沙箱策略
- [ ] 进入标准含代码审查、单测覆盖率、冒烟测试等可验证条件
- [ ] 退出标准含缺陷分级关闭要求（P0 全关闭）
- [ ] 回归范围基于 PRD/ADD 影响面评估，每项标注关联影响点
- [ ] 章节结构与 `tdd-template.md` 六章结构完全一致，无删除章节
- [ ] frontmatter 八个字段完整，含 `mvp_phase`
- [ ] TDD 中无可执行测试代码或自动化脚本
- [ ] 质量门禁清单已逐项勾选
