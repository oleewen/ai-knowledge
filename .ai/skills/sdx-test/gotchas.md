# sdx-test 常见陷阱

---

## 前置与输入

**无 PRD 直接写 TDD**：PRD 是硬输入，缺失则终止并提示先执行 `sdx-prd`。没有 PRD，用例就没有 US-n / FR-n 锚点，评审与验收时无法判断测试是否覆盖了正确的需求，测试做了也等于没做。

**无 ADD 仍按「完整接口覆盖」编造路径**：ADD 不存在时，在 TDD 概述中显式标注「缺少 ADD 基线」，接口类用例收缩为 PRD 已声明的业务边界；待 ADD 补齐后再扩展 TC-API-*。凭印象假设 API 路径与错误码，会导致用例与实际接口不符，联调时才发现测试设计需要大幅返工。

---

## 步骤 1：测试策略与范围分析

**回归范围写「全量回归」或泛泛列举模块**：§1.2 回归范围须与 ADD 影响面分析对齐，标注直接影响/间接影响与优先级（P0 核心链优先）。写「全量回归」意味着没有做影响面分析，实际执行时要么漏测关键路径，要么浪费大量时间测不相关的功能。

**忽略测试风险识别**：步骤 1 必须识别复杂度、数据一致性、并发安全、外部依赖等风险维度。跳过风险识别，后续用例设计就没有重点，容易在低风险区域过度设计，在高风险区域（如分布式锁、幂等性）覆盖不足。

---

## 步骤 2：测试用例设计

**臆测用例、无 US/API/BR 锚点**：每个 TC 至少关联 US-n、ADD 中 API 规约、或 BR-n 之一；做不到则标为待澄清项，不填空或泛写。没有锚点的用例无法在评审时验证覆盖完整性，也无法在需求变更时判断哪些用例需要同步更新。

**仅 happy path，忽略异常与边界**：每个用户故事须显式覆盖备选场景、异常场景、边界场景。只测正常路径，上线后缺陷会集中出现在异常场景——这些场景在生产环境中并不罕见，而且往往影响最大。

**§2.4 / §2.5 留空不写「不适用」**：异常场景（§2.4）和性能用例（§2.5）若无 NFR 要求或不在本次 MVP 范围，须显式写「不适用」及原因，不能留空。空章节会让评审者无法判断是遗漏还是有意跳过，质量门禁也会报错。

**忽略 `--depth` 参数**：`quick` 仅设计 P0 功能用例与核心接口用例；`standard` 完整六类用例；`deep` 额外增加性能与安全用例。不按 depth 执行，`quick` 场景下展开完整矩阵成本过高，`deep` 场景下过薄则覆盖不足，都会让用户对产出不满意。

**接口用例缺少幂等性与并发场景**：接口测试用例必须覆盖重复请求同一操作（幂等性）和同一资源并发操作（并发控制）。这两类场景是接口设计中最容易出问题的地方，但也是最容易被遗漏的——因为单次手工测试不会触发这些问题。

**业务规则用例缺少规则组合/冲突场景**：对每个 BR-n 不只设计单条规则触发，还要设计规则组合与冲突场景。多条规则同时触发时的行为往往是 bug 的高发区，单独测每条规则通过不代表组合场景也正确。

---

## 步骤 3：测试数据与环境规划

**测试数据规划缺少边界值与特殊数据**：数据需求分析须包含边界值（最大/最小/临界值）、极端值、异常数据（空值、超长、特殊字符）。只准备正常数据，边界场景的用例就无法执行，等于这些用例形同虚设。

**外部依赖无 Mock/Stub 策略**：涉及第三方 RPC、MQ 消息的测试场景，必须明确 Mock/Stub 策略（哪些依赖需要 Mock、Mock 的行为是什么）。没有 Mock 策略，测试环境搭建时才发现依赖不可用，导致测试阻塞。

---

## 步骤 4：进出标准与回归策略

**进出标准不可度量**：进入/退出标准必须是可检查的具体条件（如「P0/P1 用例通过率 100%」「无未解决 P0 缺陷」），不能写「测试完成」「质量达标」等模糊表述。不可度量的标准在实际执行时无法判断是否满足，导致发布决策依赖主观判断。

**回归策略与影响面脱节**：回归执行顺序须基于 ADD 影响面分析（核心流程 → 直接影响 → 间接影响），不能简单列举模块名。没有优先级的回归策略，时间紧张时无法做出合理取舍，容易漏测高风险路径。

---

## 步骤 5：文档输出

**在 TDD 中写自动化脚本或可执行测试代码**：仅产出 `TDD-*.md`；自动化实现留在开发流程与代码仓。把代码写进 TDD 会混淆设计阶段与实现阶段，模板章节被代码挤占，且代码变更时 TDD 难以同步维护。

**跳过或重排模板章节**：严格遵循 [assets/tdd-template.md](assets/tdd-template.md) 六章结构；无内容章节保留标题并写「不适用」或「待补充」，不删除。`validate-test.sh` 与质量门禁会检查章节完整性，章节缺失会直接导致门禁失败。

**文档元数据位置错误或字段缺失**：勿在文件开头写 YAML frontmatter；仅在文末「## 文档元数据」fenced `yaml` 中填写；须包含 `id`、`title`、`version`、`status`、`created`、`updated`、`parent`、`mvp_phase`，初始 `status` 为 `draft`，`parent` 填写上游 PRD 编号。

**未执行质量门禁自查即宣称完成**：步骤 5 必须逐项检查 [reference/quality-checklist.md](reference/quality-checklist.md)；必要时运行 `validate-test.sh`。跳过自查，frontmatter、子章节或追溯字段遗漏会流入下游，在测试执行阶段才被发现，修复成本更高。

---

## 快速自查清单

- [ ] PRD 文档已落盘且可用，`parent` 与 `mvp_phase` 字段已填写
- [ ] ADD 缺失时已在概述中标注，接口用例已收缩为 PRD 边界
- [ ] 测试范围已划定新增/变更/回归三类，风险区域已识别
- [ ] 每个 TC 关联 US-n / API 规约 / BR-n，无孤立用例
- [ ] 功能用例覆盖正常场景与关键异常/边界场景
- [ ] 接口用例覆盖幂等性与并发控制场景
- [ ] 业务规则用例覆盖规则组合与冲突场景
- [ ] §2.4 异常场景已覆盖（或显式标注「不适用」及原因）
- [ ] §2.5 性能用例已覆盖（或显式标注「不适用」及原因）
- [ ] §2.6 回归范围与 ADD 影响面对齐，有优先级排序
- [ ] 测试数据含边界值与特殊数据，外部依赖有 Mock/Stub 策略
- [ ] 进出标准可度量、可检查，非模糊表述
- [ ] 六章结构完整，无删除章节
- [ ] 文末元数据 YAML 字段完整（含 `parent`、`mvp_phase`）；文件开头无 `---` YAML
- [ ] TDD 中无自动化测试代码
- [ ] quality-checklist.md 已逐项勾选
