# 测试设计质量验收清单（sdx-test）

步骤 5 验收用：Agent 在文档输出与评审时复制本清单并逐项勾选。原则与反模式见 [design-principles.md](design-principles.md)；模板结构见 [../assets/tdd-template.md](../assets/tdd-template.md)。

## 完整性

- [ ] 测试目标（§1.1）：关联 PRD 与 ADD 明确
- [ ] 测试范围（§1.2）：新增/变更/回归范围均已列出
- [ ] 测试策略（§1.3）：单元/集成/端到端覆盖目标明确
- [ ] 功能测试用例（§2.1）：覆盖所有用户故事的验收标准
- [ ] 接口测试用例（§2.2）：覆盖所有 API 的正常与异常路径
- [ ] 业务规则测试用例（§2.3）：覆盖所有 BR-n 的触发与边界条件
- [ ] 异常场景测试用例（§2.4）：系统异常/数据异常/并发异常已覆盖
- [ ] 性能测试用例（§2.5）：非功能需求有对应用例（或标注「不适用」）
- [ ] 回归测试用例（§2.6）：基于影响面分析确定回归范围
- [ ] 测试数据（§3）：每类数据有准备方式与数据量说明
- [ ] 测试环境（§4）：服务/数据库/中间件/外部依赖要求完整
- [ ] 进入标准（§5.1）：代码/单测/环境/数据就绪条件明确
- [ ] 退出标准（§5.2）：用例通过率/缺陷/覆盖率标准明确

## 一致性

- [ ] 功能用例 TC-{NNN} 与 PRD 用户故事 US-{NNN} 一一对应
- [ ] 接口用例 TC-API-{NNN} 与 ADD 中 API 规约一致
- [ ] 业务规则用例 TC-BR-{NNN} 与需求分析 BR-{NNN} 一致
- [ ] 文档内部 TC-* 编号连续无断号
- [ ] 覆盖率目标与测试策略中的声明一致
- [ ] 优先级标注与 PRD 用户故事优先级对齐

## 可行性

- [ ] 测试数据可通过指定方式准备
- [ ] 测试环境可部署且依赖可满足
- [ ] 外部依赖有 Mock / Stub 策略
- [ ] 测试工期在项目计划范围内

## 可追溯性

- [ ] 每个 TC-{NNN} 可追溯到 US-{NNN}
- [ ] 每个 TC-API-{NNN} 可追溯到 API 规约
- [ ] 每个 TC-BR-{NNN} 可追溯到 BR-{NNN}
- [ ] 每个 TC-REG-{NNN} 可追溯到影响面分析
- [ ] 每个 TC-EX-{NNN} 可追溯到异常处理设计

## 格式规范

- [ ] 文末「## 文档元数据」内 fenced YAML 完整（id、title、version、status、created、updated、parent、mvp_phase）；**文件开头无** `---` YAML frontmatter
- [ ] 章节结构与 tdd-template.md 一致，编号连续
- [ ] 表格格式规范，列宽合理
- [ ] 变更历史（§6.1）记录初始版本
