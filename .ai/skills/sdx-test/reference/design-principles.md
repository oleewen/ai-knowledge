# 设计原则与反模式

sdx-test 技能的完整设计约束。主文件 SKILL.md 中的「核心约束」为精简版，本文件为完整规范。

---

## 设计原则

### 1. 模板驱动

输出严格遵循 [../assets/tdd-template.md](../assets/tdd-template.md) 的六章结构（概述→测试用例→测试数据→测试环境→测试进出标准→附录）。章节顺序与编号不可调整；无内容的章节保留标题并标注「不适用」或「待补充」。

### 2. 证据优先

测试用例设计须引用上游文档事实，禁止凭空臆测。引用格式遵循项目规范：

| 证据类型 | 格式 | 示例 |
|----------|------|------|
| 用户故事 | `US-{NNN}` | `US-001 创建申诉单` |
| 功能需求 | `FR-{NNN}` | `FR-001 申诉单创建` |
| 业务规则 | `BR-{NNN}` | `BR-001 申诉时限校验` |
| API 规约 | `{Method} {Path}` | `POST /api/appeal/create` |
| 知识库实体 | `{视角}-{ID}` | `BC-001 FeeAppealContext` |
| 代码位置 | `{类}:{方法}` | `PolicyAppealApi:create` |

### 3. 按需加载

仅在本轮任务需要时打开文件：

- 步骤 1 读 PRD 用户故事清单 + ADD 架构概览
- 步骤 2 按测试类型打开对应规约（API 规约 → 接口用例、业务规则 → 规则用例）
- 步骤 3 打开环境配置与数据模型相关文档
- 步骤 4 综合已读内容制定进出标准

禁止为「完整性」通读 `knowledge/**` 或全仓源码。

### 4. 歧义标注

对 PRD 或 ADD 中验收标准不明确、接口定义不完整、业务规则边界模糊的情况，标注为待澄清项，在测试设计文档中显式记录。禁止自行假设缺失信息后继续，应暂停并请求用户澄清。

### 5. 范围边界

本技能仅负责输出测试设计文档。不产出自动化测试代码、测试脚本或测试报告——自动化测试代码在开发阶段产出，测试报告在执行阶段产出。

### 6. 可追溯性

- 每个功能测试用例（TC-{NNN}）可追溯到用户故事（US-{NNN}）
- 每个接口测试用例（TC-API-{NNN}）可追溯到 API 规约
- 每个业务规则测试用例（TC-BR-{NNN}）可追溯到业务规则（BR-{NNN}）
- 每个回归测试用例（TC-REG-{NNN}）可追溯到影响面分析
- 每个异常场景测试用例（TC-EX-{NNN}）可追溯到异常处理设计

### 7. ID 编号规范

测试设计文档使用以下 ID 编号体系：

| 前缀 | 用途 | 示例 |
|------|------|------|
| `TDD-{REQUIREMENT-ID}-MVP{N}` | 文档编号 | `TDD-20260327-001-MVP1` |
| `TC-{NNN}` | 功能测试用例 | `TC-001`、`TC-002` |
| `TC-API-{NNN}` | 接口测试用例 | `TC-API-001` |
| `TC-BR-{NNN}` | 业务规则测试用例 | `TC-BR-001` |
| `TC-EX-{NNN}` | 异常场景测试用例 | `TC-EX-001` |
| `TC-PERF-{NNN}` | 性能测试用例 | `TC-PERF-001` |
| `TC-REG-{NNN}` | 回归测试用例 | `TC-REG-001` |

### 8. 覆盖率目标

默认覆盖率目标（可在步骤 1 中根据项目实际调整）：

| 测试层次 | 覆盖指标 | 目标值 |
|----------|----------|--------|
| 单元测试 | 行覆盖率 | ≥ 80% |
| 集成测试 | 接口覆盖率 | 100% |
| 端到端测试 | 核心场景覆盖率 | 100% |

## 反模式清单（禁止）

| 反模式 | 说明 |
|--------|------|
| 臆测用例 | 未引用 PRD 用户故事或 ADD 接口即自行编造测试用例 |
| 吞没歧义 | 将验收标准不明确的用户故事自行补全后设计用例，不标注待澄清 |
| 范围蔓延 | 在测试设计中编写自动化测试代码或生成测试报告 |
| 模板跳章 | 跳过或重排 tdd-template.md 的章节结构 |
| 通读全库 | 一次性读取全仓知识库而不考虑本轮任务实际需要 |
| 无编号用例 | 测试用例无统一编号前缀，导致不可追溯 |
| 孤立用例 | 测试用例未关联到用户故事、API 或业务规则 |
| 忽略异常 | 仅设计正常场景用例，忽略异常/边界/并发场景 |
| 空章提交 | 章节标题下无任何内容且未标注「不适用」或「待补充」 |
| 忽略回归 | 未根据影响面分析制定回归测试范围 |

## 质量标准

著文层面的目标：策略覆盖各测试层次与新增/变更/回归；用例与 PRD/ADD 一致且编号可追溯；数据、环境与进出标准可落地。

> 完整验收清单见 [quality-checklist.md](quality-checklist.md)。

## 错误处理

| 错误场景 | 处理方式 |
|----------|----------|
| PRD 文档不存在 | 终止，提示先执行 sdx-prd |
| ADD 文档不存在 | 发出警告，仅基于 PRD 完成测试设计，标注缺少技术设计基线 |
| PRD 中用户故事验收标准不完整 | 标注为待澄清项，基于已有信息设计用例并备注限制 |
| knowledge 目录缺失 | 发出警告，仅基于 PRD 与 ADD 完成设计 |
| 模板文件不存在 | 终止，提示创建 `.ai/skills/sdx-test/assets/tdd-template.md` |
| 输出目录不存在 | 自动创建 `system/requirements/REQUIREMENT-{YYMMDD}-{主题slug}/MVP-Phase-{N}/` 目录 |
