# document-indexing 常见陷阱（Gotchas）

本文记录执行 `document-indexing` Skill 时高频踩坑点，供 Agent 与人类开发者参考。

---

## 1. 模式与深度选择

### 1.1 无历史日志仍强行增量模式
**陷阱**：`indexing-log.jsonl` 不存在或为空，Agent 仍尝试以增量模式运行。  
**后果**：无基线时间戳，扫描范围不确定，产出索引残缺。  
**正确做法**：无历史日志时强制降级为 full 模式，并提示用户。

### 1.2 增量基线时间戳来源错误
**陷阱**：从 `changes-index.json` 的 `baseline_time` 而非 `indexing-log.jsonl` 的 `indexing_finished_at` 读取基线。  
**后果**：基线偏移，漏扫或重复扫描文件。  
**正确做法**：增量模式的基线时间戳唯一来源是 `indexing-log.jsonl` 最后一条记录的 `indexing_finished_at`。

### 1.3 深度选错导致信息不足或过载
**陷阱**：大型仓库初扫直接用 depth=3，或日常增量更新用 depth=1。  
**后果**：depth=3 全量精读耗时极长；depth=1 增量更新丢失结构细节。  
**正确做法**：大型仓库初扫用 depth=1 快速概览，日常增量用 depth=2，逆向工程或全量索引才用 depth=3。

---

## 2. 扫描与采集阶段

### 2.1 索引构建产物目录
**陷阱**：未排除 `target/`、`build/`、`node_modules/` 等目录，将编译产物纳入索引。  
**后果**：索引膨胀，充斥无意义的 `.class`、`.jar` 条目，干扰 Agent 检索。  
**正确做法**：严格按质量标准的排除列表过滤，输出目录自身也须排除。

### 2.2 把未读文件写成已索引内容
**陷阱**：仅扫描目录结构，未实际读取文件内容，却在索引中描述文件的具体实现细节。  
**后果**：产出含幻觉信息，误导后续 Agent 和开发者。  
**正确做法**：零幻觉原则——只索引实际读取的内容；未读路径标注 `[未索引]` 并说明原因，归入 §八。

### 2.3 增量模式清空未变更章节
**陷阱**：增量更新时重写整个 INDEX_GUIDE.md，导致未变更章节内容丢失。  
**后果**：历史积累的索引内容被清空，退化为残缺索引。  
**正确做法**：增量模式只合并更新变更文件对应的条目，未变更部分保持原样。

### 2.4 同一文件归入多个章节
**陷阱**：一个文件同时出现在 §三接口清单和 §四领域模型中。  
**后果**：违反 MECE 原则，维护时需同步多处，极易失步。  
**正确做法**：每个文件只归入一个最匹配的章节，跨章节引用以超链接形式体现，不重复内容。

---

## 3. 九章结构生成

### 3.1 留有空章节
**陷阱**：某章节（如 §三消息队列、§五状态流转）在当前项目中无对应内容，直接留空或写占位符。  
**后果**：索引结构不完整，Agent 无法判断是「确实无内容」还是「未扫描到」。  
**正确做法**：确实无内容的小节标注 `[未索引]` 并说明原因（如「当前项目无消息队列组件」），归入 §八排除列表。

### 3.2 条目信息密度不达标
**陷阱**：表格行写成「用户服务」「订单模块」等空洞标签，无实质描述。  
**后果**：索引可检索性差，Agent 无法从条目判断是否需要深入阅读。  
**正确做法**：每条目 15-30 字，包含功能描述、关键属性或路径，禁止空话套话。

### 3.3 路径使用绝对路径或错误相对路径
**陷阱**：索引中出现 `/Users/xxx/project/src/...` 或 `../src/...` 形式的路径。  
**后果**：路径在其他环境下不可访问，交叉引用失效。  
**正确做法**：一律使用项目根相对路径，如 `src/main/java/com/example/Service.java`。

### 3.4 版本号、配置值凭印象填写
**陷阱**：§七配置中心的版本号或配置值未从实际文件读取，凭经验或猜测填写。  
**后果**：索引与实际配置不符，误导依赖该信息的 Agent 或开发者。  
**正确做法**：版本号、配置值必须来自实际读取的 `pom.xml`、`application.yml` 等文件。

## 4. 输出与日志

### 4.1 输出路径优先级混淆
**陷阱**：用户未指定 `--output` 时，直接输出到项目根目录 `INDEX_GUIDE.md`，跳过 `system/` 检查。  
**后果**：与约定路径不符，下游 agent-guide 等 Skill 按优先级查找时找不到文件。  
**正确做法**：严格按优先级：用户指定 > `./system/` > `./doc/` > `./INDEX_GUIDE.md`。

### 4.2 未追加操作日志
**陷阱**：索引生成完毕后未写入 `indexing-log.jsonl`，或覆盖而非追加。  
**后果**：下次增量模式无法获取基线时间戳，被迫降级为 full 模式；历史记录丢失。  
**正确做法**：每次执行完毕必须追加（append）一条完整日志记录，包含 `indexing_finished_at`。

### 4.3 日志字段缺失
**陷阱**：日志记录缺少 `data_mode`、`read_mode`、`indexed_files` 等字段。  
**后果**：下游 Skill 解析日志失败，增量基线计算出错。  
**正确做法**：严格按日志格式输出所有必需字段，包括 `timestamp`、`data_mode`、`read_mode`、`output_path`、`indexed_files`、`duration_ms`、`indexing_finished_at`。

---

## 5. 与上下游 Skill 的协作

### 5.1 在本 Skill 内跳过 document-change 直接扫描
**陷阱**：增量模式下未调用 `document-change` 获取变更文件列表，直接扫描全部文件。  
**后果**：增量模式退化为全量扫描，失去增量价值。  
**正确做法**：增量模式必须先通过 `document-change` 获取 `changes-index.json`，以变更文件列表驱动扫描范围。

### 5.2 与 agent-guide 产出的目录树矛盾
**陷阱**：INDEX_GUIDE.md §二目录树与 README.md 中的目录树描述不一致。  
**后果**：Agent 获得矛盾的项目结构认知，影响导航准确性。  
**正确做法**：agent-guide 生成 README 时以本 INDEX §二为唯一来源；本 Skill 更新 §二后需通知 agent-guide 同步。

## 快速检查清单

执行完毕后，对照以下项目快速自查：

- [ ] 九章均有实质内容，无空章节（空内容标注 `[未索引]`）
- [ ] 每条目 15-30 字，无占位符和空话
- [ ] 所有路径为项目根相对路径，可访问
- [ ] 版本号、配置值来自实际读取的文件
- [ ] 每个文件只归入一个章节（MECE）
- [ ] 操作日志已追加到 `indexing-log.jsonl`，字段完整
- [ ] 增量模式未清空未变更章节
- [ ] 构建产物目录已排除
- [ ] 未读内容未写成已核实事实（零幻觉）
