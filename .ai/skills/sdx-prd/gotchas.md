# sdx-prd 常见陷阱（Gotchas）

本文记录执行 `sdx-prd` 时高频踩坑点。入口与工作流摘要见 [SKILL.md](SKILL.md)；算法细节见 [reference/workflow-spec.md](reference/workflow-spec.md)；步骤 5 逐项验收见 [reference/quality-checklist.md](reference/quality-checklist.md)。

## 与本 Skill 其他文件的关系

| 文件 | 何时读 |
|------|--------|
| `SKILL.md` | 每次任务：IO、参数、五步摘要、约束与索引 |
| `gotchas.md`（本文件） | 开始前扫一眼；卡顿时按步骤查陷阱 |
| `assets/prd-template.md` | 撰写/整合 `PRD-{YYMMDD}-{主题slug}.md` 时 |
| `reference/workflow-spec.md` | 需明确步骤算法、决策点、步间产出时 |
| `reference/audience-and-language.md` | 不确定正文能否写某类技术词时 |
| `reference/core-concepts.md` | 核对术语口径时 |
| `reference/design-principles.md` | 编号体系、反模式、错误处理 |
| `reference/quality-checklist.md` | 定稿前门禁 |
| `scripts/validate-prd.sh` | 本地校验结构时 |

## 1. 输入与前置条件

### 1.1 无需求分析文档直接开始编写 PRD
**陷阱**：用户描述业务需求，Agent 跳过 `sdx-analysis` 直接输出 PRD。  
**后果**：缺少 FR-n / BR-n 编号作为追溯锚点，用户故事无法追溯到业务目标，可追溯性链条从根部断裂。  
**正确做法**：需求分析文档（`system/analysis/ANALYSIS-{ID}.md`）是硬输入；不存在时终止并提示先执行 `sdx-analysis`。

### 1.2 未限定目标 MVP 范围
**陷阱**：未读取需求分析文档中的 MVP 拆分章节，将所有 MVP 的功能需求一并纳入 PRD。  
**后果**：PRD 超越 MVP 边界，引入后续 MVP 的功能，导致范围蔓延，下游 `sdx-design` 过度设计。  
**正确做法**：严格聚焦 `--mvp` 参数指定的 MVP 范围；需求分析文档中该 MVP 未包含的 FR-n 一律不纳入本次 PRD。

### 1.3 需求分析文档结构不完整时静默继续
**陷阱**：需求分析文档缺少 FR-n 优先级或 BR-n 独立编号，Agent 未发出警告直接推进。  
**后果**：基于残缺输入生成的 PRD 同样残缺，用户故事验收标准无法追溯，且用户无感知。  
**正确做法**：发出警告并列出缺失内容，在文档中显式标注「基于不完整需求分析，存在 PRD 盲区」。

## 2. 步骤 1：业务流程设计

### 2.1 主流程与分支/异常流程混写
**陷阱**：将主成功路径、备选路径、异常路径混在同一 Mermaid 图中，未分层描述。  
**后果**：流程图过于复杂，评审时难以理解主干逻辑；下游 `sdx-design` 无法清晰识别核心路径。  
**正确做法**：主流程单独绘制，分支/异常流程以扩展节点或独立子图补充；每条路径标注触发条件。

### 2.2 跨系统交互未标注系统边界
**陷阱**：流程图中涉及外部系统调用，但未标注系统名称、交互协议或异步/同步属性。  
**后果**：下游 `sdx-design` 无法判断集成方式，接口设计缺少依据，集成风险被低估。  
**正确做法**：每个跨系统交互节点须标注：系统名称、交互方式（同步/异步）、关键输入输出；异步回调须单独标注回调路径。

### 2.3 业务规则内嵌流程节点描述
**陷阱**：在流程图节点文字中直接写业务规则（如「金额 > 1000 需审批」），未引用 BR-n。  
**后果**：业务规则无独立编号，无法被多个流程复用引用，也无法单独追溯到需求分析 BR-n。  
**正确做法**：流程节点只描述动作，业务规则独立编号为 BR-n，在节点旁标注 `[BR-n]` 引用。

## 3. 步骤 2：用例建模

### 3.1 用例图参与者与实际角色不符
**陷阱**：用例图中的参与者（Actor）使用技术角色（如「系统」「服务」），而非业务角色（如「运营人员」「商家」）。  
**后果**：用例图无法反映真实业务场景，业务方评审时无法理解，失去用例建模的沟通价值。  
**正确做法**：参与者须使用业务角色名称；系统间交互以「外部系统」作为参与者，标注系统名称。

### 3.2 用例描述缺少前后置条件
**陷阱**：用例描述只有主成功场景，未定义前置条件（Precondition）和后置条件（Postcondition）。  
**后果**：下游 `sdx-design` 无法确定接口调用的合法状态，状态机设计缺少边界约束。  
**正确做法**：每个用例必须包含前置条件（触发用例前系统/数据须满足的状态）和后置条件（用例成功后系统/数据的状态变化）。

### 3.3 用户故事与用例未建立双向映射
**陷阱**：用户故事和用例分别独立编写，未建立 US-n ↔ UC-n 的映射关系。  
**后果**：两套模型各自孤立，评审时无法交叉验证覆盖完整性，遗漏场景难以发现。  
**正确做法**：在用例描述中标注关联的用户故事编号（US-n），在用户故事中标注关联的用例编号（UC-n），确保双向映射完整。

## 4. 步骤 3：用户故事与场景

### 4.1 用户故事缺少验收标准
**陷阱**：用户故事只有「作为…我希望…以便…」三段式，未编写 Given-When-Then 验收标准。  
**后果**：验收标准缺失，测试设计无依据，交付时无法判断故事是否完成，验收扯皮。  
**正确做法**：每个用户故事必须包含至少一条 Given-When-Then 验收标准，覆盖正常场景；边界/异常场景须额外补充。

### 4.2 仅覆盖正常场景，遗漏边界与异常
**陷阱**：用户故事只描述「一切顺利」的主路径，未考虑输入边界、并发冲突、外部依赖失败等场景。  
**后果**：PRD 覆盖不完整，下游 `sdx-design` 遗漏异常处理设计，上线后出现未预期的错误行为。  
**正确做法**：每个用户故事须显式列出备选场景（Alternative）、异常场景（Exception）、边界场景（Boundary），并各自编写验收标准。

### 4.3 用户故事无法独立交付（违反 INVEST 原则）
**陷阱**：用户故事之间存在强依赖，US-2 必须与 US-1 同时交付才有业务价值。  
**后果**：故事无法独立演示，Sprint 计划无法灵活调整，交付粒度过粗。  
**正确做法**：每个用户故事须能独立演示给业务方；发现强依赖时合并为一个故事或拆分为独立可交付单元。

### 4.4 用户故事未关联 FR-n 与 BR-n
**陷阱**：用户故事只有描述，未标注关联的功能需求编号（FR-n）和业务规则编号（BR-n）。  
**后果**：PRD 与需求分析文档的追溯链断裂，无法验证 PRD 是否完整覆盖了需求分析的功能需求。  
**正确做法**：每个用户故事必须标注 `关联 FR-n` 和 `关联 BR-n`，确保双向可追溯。

## 5. 步骤 4：功能模块与交互设计

### 5.1 功能模块划分按技术层次而非业务能力
**陷阱**：功能模块按「前端模块」「后端模块」「数据库模块」划分，而非按业务能力域划分。  
**后果**：模块边界与业务边界不一致，下游 `sdx-design` 的服务拆分与 PRD 模块无法对应。  
**正确做法**：功能模块按业务能力域划分（如「报价管理」「审批流程」「通知中心」），每个模块描述其业务职责与对外接口。

### 5.2 业务规则汇总缺少优先级与冲突处理
**陷阱**：业务规则只列出触发条件和执行逻辑，未标注规则优先级，也未说明规则冲突时的处理策略。  
**后果**：多条规则同时触发时行为不确定，开发阶段各自实现，上线后出现不一致行为。  
**正确做法**：每条 BR-n 必须标注优先级（数字越小优先级越高）；存在互斥规则时须显式说明冲突处理策略。

### 5.3 数据字典缺少字段约束与枚举值
**陷阱**：数据字典只列出字段名和类型，未定义长度限制、非空约束、枚举值范围。  
**后果**：前端校验与后端校验标准不一致，数据库约束设计缺少依据，数据质量问题在上线后才暴露。  
**正确做法**：数据字典每个字段须包含：字段名、类型、长度/精度、是否必填、枚举值（如适用）、业务含义说明。

### 5.4 状态机缺少非法状态转换说明
**陷阱**：状态机只绘制合法的状态转换路径，未说明哪些转换是非法的及其处理方式。  
**后果**：下游 `sdx-design` 遗漏非法状态转换的防护逻辑，运行时出现状态异常。  
**正确做法**：状态机须显式标注终态（Terminal State）；对关键非法转换（如「已完成 → 处理中」）须说明拦截策略。

## 6. 步骤 5：文档输出与评审

### 6.1 跳过或重排模板章节
**陷阱**：认为某章节「不适用」直接删除，或调整章节顺序以「更符合逻辑」。  
**后果**：下游 `sdx-design` 按固定章节引用 PRD，章节缺失或错位导致引用失效。  
**正确做法**：章节结构与 `prd-template.md` 十章结构完全一致；无内容的章节保留标题并标注「不适用」或「待补充」，不得删除。

### 6.2 frontmatter 字段缺失
**陷阱**：输出文档缺少 `parent`（关联需求分析编号）、`mvp_phase` 或 `status` 字段。  
**后果**：文档与上游需求分析的追溯链断裂；`mvp_phase` 缺失导致多 MVP 场景下管理混乱。  
**正确做法**：frontmatter 必须包含 `id`、`title`、`version`、`status`、`created`、`updated`、`parent`、`mvp_phase` 八个字段，初始 `status` 为 `draft`。

### 6.3 在 PRD 中写入技术实现细节
**陷阱**：在功能模块设计或业务规则中写出技术方案（如「使用 Redis 实现分布式锁」「调用 XXX 微服务接口」）。  
**后果**：范围蔓延，PRD 与 ADD 职责混淆，技术方案变更时 PRD 需同步修改，维护成本高。  
**正确做法**：PRD 只描述「做什么」和「业务规则」，不描述「怎么做」；技术实现细节留给下游 `sdx-design`。

### 6.4 吞没歧义自行假设
**陷阱**：需求分析文档中某 FR-n 描述模糊，Agent 自行假设后继续编写用户故事和用例。  
**后果**：PRD 基于错误假设，下游设计方案全部偏离，评审时才发现，返工成本高。  
**正确做法**：歧义项标注为「待澄清」，在文档中显式记录疑问点，暂停并请求用户澄清后再继续。

### 6.5 Mermaid 图表语法错误
**陷阱**：业务流程图、用例图、状态机使用了不合法的 Mermaid 语法，无法渲染。  
**后果**：文档在 GitHub / IDE 中显示为原始文本，可视化失效，评审时无法直观理解业务流程。  
**正确做法**：输出前验证 Mermaid 语法；流程图用 `flowchart`，时序图用 `sequenceDiagram`，状态机用 `stateDiagram-v2`，用例图用 `graph`。

### 6.6 跳过质量门禁自查
**陷阱**：PRD 写完后直接输出，未逐项检查 `reference/quality-checklist.md`。  
**后果**：编号断号、追溯链断裂、章节缺失等问题流入下游，被 `sdx-design` 或评审发现时返工成本高。  
**正确做法**：步骤 5 必须执行质量门禁自查，所有检查项通过后才输出文档。
