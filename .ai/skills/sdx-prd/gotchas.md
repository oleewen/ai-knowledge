# sdx-prd 常见陷阱

---

## 输入与前置

**无需求分析文档直接开始编写 PRD**：需求分析文档（`system/analysis/ANALYSIS-{ID}.md`）是硬输入；不存在时终止并提示先执行 `sdx-analysis`。US-n 无法追溯到 FR-n 时，可追溯性链条从根部断裂。

**未限定目标 MVP 范围**：严格聚焦 `--mvp` 参数指定的 MVP 范围；需求分析文档中该 MVP 未包含的 FR-n 一律不纳入本次 PRD，否则引入后续 MVP 的功能导致范围蔓延。

**需求分析文档结构不完整时静默继续**：发出警告并列出缺失内容，在文档中显式标注「基于不完整需求分析，存在 PRD 盲区」。

---

## 步骤 1：业务流程设计

**主流程与分支/异常流程混写**：主流程单独绘制，分支/异常流程以扩展节点或独立子图补充；每条路径标注触发条件。混写导致流程图过于复杂，评审时难以理解主干逻辑。

**跨系统交互未标注系统边界**：每个跨系统交互节点须标注系统名称、交互方式（同步/异步）、关键输入输出；异步回调须单独标注回调路径，否则下游 sdx-design 无法判断集成方式。

**业务规则内嵌流程节点描述**：流程节点只描述动作，业务规则独立编号为 BR-n，在节点旁标注 `[BR-n]` 引用，不在节点文字中直接写规则内容。

---

## 步骤 2：用例建模

**用例图参与者与实际角色不符**：参与者须使用业务角色名称（如「运营人员」「商家」），不用技术角色（如「系统」「服务」）；系统间交互以「外部系统」作为参与者并标注系统名称。

**用例描述缺少前后置条件**：每个用例必须包含前置条件（触发前系统/数据须满足的状态）和后置条件（成功后系统/数据的状态变化），否则下游 sdx-design 无法确定接口调用的合法状态。

**用户故事与用例未建立双向映射**：在用例描述中标注关联的 US-n，在用户故事中标注关联的 UC-n，确保双向映射完整，否则两套模型各自孤立，无法交叉验证覆盖完整性。

---

## 步骤 3：用户故事与场景

**用户故事缺少验收标准**：每个用户故事必须包含至少一条 Given-When-Then 验收标准，覆盖正常场景；边界/异常场景须额外补充，否则交付时无法判断故事是否完成。

**仅覆盖正常场景，遗漏边界与异常**：每个用户故事须显式列出备选场景、异常场景、边界场景，并各自编写验收标准，否则下游 sdx-design 遗漏异常处理设计。

**用户故事无法独立交付（违反 INVEST 原则）**：每个用户故事须能独立演示给业务方；发现强依赖时合并为一个故事或拆分为独立可交付单元。

**用户故事未关联 FR-n 与 BR-n**：每个用户故事必须标注 `关联 FR-n` 和 `关联 BR-n`，确保 PRD 与需求分析文档的追溯链完整。

---

## 步骤 4：功能模块与交互设计

**功能模块划分按技术层次而非业务能力**：功能模块按业务能力域划分（如「报价管理」「审批流程」），不按「前端模块」「后端模块」划分，否则模块边界与业务边界不一致。

**业务规则汇总缺少优先级与冲突处理**：每条 BR-n 必须标注优先级；存在互斥规则时须显式说明冲突处理策略，否则多条规则同时触发时行为不确定。

**数据字典缺少字段约束与枚举值**：数据字典每个字段须包含类型、长度/精度、是否必填、枚举值（如适用）、业务含义说明，否则前后端校验标准不一致。

**状态机缺少非法状态转换说明**：状态机须显式标注终态；对关键非法转换须说明拦截策略，否则下游 sdx-design 遗漏防护逻辑。

---

## 步骤 5：文档输出

**跳过或重排模板章节**：章节结构与 `prd-template.md` 十章结构完全一致；无内容的章节保留标题并标注「不适用」或「待补充」，不删除。下游 sdx-design 按固定章节引用，章节缺失或错位导致引用失效。

**文档元数据字段缺失或位置错误**：勿在文件开头写 YAML frontmatter；仅在文末「## 文档元数据」fenced `yaml` 中填写字段；须包含 `id`、`title`、`version`、`status`、`created`、`updated`、`parent`、`mvp_phase`，初始 `status` 为 `draft`，`parent` 填写上游需求分析编号。

**在 PRD 中写入技术实现细节**：PRD 只描述「做什么」和「业务规则」，不描述「怎么做」；「使用 Redis 实现分布式锁」「调用 XXX 微服务接口」等技术细节留给下游 sdx-design。

**吞没歧义自行假设**：歧义项标注为「待澄清」，暂停并请求用户澄清后再继续，不自行假设后继续编写。

**Mermaid 图表语法错误**：输出前验证 Mermaid 语法；流程图用 `flowchart`，时序图用 `sequenceDiagram`，状态机用 `stateDiagram-v2`，用例图用 `graph`。

**跳过质量门禁自查**：步骤 5 必须逐项检查 [reference/quality-checklist.md](reference/quality-checklist.md)，所有检查项通过后才输出文档。

## 快速自查清单

- [ ] 需求分析文档已落盘且可用，`parent` 与 `mvp_phase` 字段已填写
- [ ] PRD 范围严格限定在 `--mvp` 指定的 MVP，无越界功能
- [ ] 每个 US-n 关联 FR-n 和 BR-n，双向可追溯
- [ ] 每个 UC-n 有前后置条件，与 US-n 双向映射
- [ ] 每个用户故事有 Given-When-Then 验收标准（含异常场景）
- [ ] 功能模块按业务能力域划分，非技术层次
- [ ] 业务规则集中汇总在 §7，有优先级与冲突处理
- [ ] 数据字典字段完整（类型、约束、枚举值）
- [ ] 状态机标注终态与非法转换拦截策略
- [ ] 十章结构完整，无删除章节
- [ ] 文末元数据 YAML 字段完整（含 `parent`、`mvp_phase`）；文件开头无 `---` YAML
- [ ] 正文无技术实现细节（接口名、表名、中间件等）
- [ ] Mermaid 图表语法正确
- [ ] quality-checklist.md 已逐项勾选
