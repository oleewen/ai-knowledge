# PRD 质量门禁清单

步骤 5 验收用。Agent 在文档输出与评审时复制本清单并逐项勾选。

产出结构须与 [../assets/prd-template.md](../assets/prd-template.md) 十一章一致。

## 完整性

- 产品概述（§1）：产品目标引用解决方案与需求分析文档及 MVP；**§1.2 成功标准与价值度量**已从 ANALYSIS 摘录或标「待澄清」；**§1.3** 含本 MVP 范围与 **Growth/Vision 后续阶段表**（无则标「不适用」）；**§1.4** 涉及角色完整
- 核心业务流程（§2.1）：主流程 Mermaid 图完整，步骤说明表齐全
- 分支异常流程（§2.2）：异常场景 EX-n 清单完整，处理方式与用户提示明确
- 系统交互流程（§2.3）：跨系统调用时序图完整（或标注「不适用」）
- 产品交互（§3）：交互流程与交互设计说明完整
- 用例图（§4.1）：覆盖所有角色与核心交互
- 用例详述（§4.2）：每个 UC-n 有参与者、前后置条件、主成功场景、扩展场景
- 用户故事清单（§5.1）：MVP 范围内每个 FR-n 至少有一个 US-n
- 用户故事详述（§5.2）：每个 US-n 有验收标准（Given-When-Then），至少含正常+异常场景
- 功能模块（§6）：模块划分完整，模块间关系图清晰
- 业务规则汇总（§7）：所有 BR-n 集中汇总，字段完整
- 数据字典（§8）：业务术语与状态定义完整
- 非功能需求（§9）：**NFR**（选择性类别 + 可度量表述）与 ANALYSIS 对齐
- 验收标准汇总（§10）：**§10.1** 功能验收 AC-n；**§10.2** 非功能验收 NAC-n 与 §9 可互链

## 一致性

- US-n 与需求分析 FR-n 一一对应，无遗漏
- BR-n 与需求分析 BR-n 及 `system/knowledge/business/` 无矛盾
- 业务流程与需求分析中的核心场景描述一致
- US-n / UC-n / BR-n / EX-n / AC-n / NAC-n 编号连续无断号
- 各章节交叉引用自洽（故事→用例→规则→验收）
- 用例与用户故事双向映射完整

## 可行性

- 用户故事粒度合理，可在单迭代内交付
- 业务流程无逻辑死循环或不可达状态
- 验收标准可度量、可测试
- 交互设计在现有技术框架下可实现

## 可测性与实现边界（与 BMAD 质检维度对齐）

- **无实现泄漏**：需求表述不以具体技术栈、中间件、库、表结构作为验收依据（与 [design-principles.md](design-principles.md) 一致）
- **关键句可测**：成功标准、AC、NFR/NAC 含可验证口径，避免仅主观形容词或模糊量词而无验收口径
- **（可选）领域关键词**：强监管或行业合规场景下，核对 ANALYSIS 与 §9 是否覆盖必要合规/安全关键词（无则标风险或待澄清）

## 可追溯性

- 每个 US-n 可追溯到需求分析 FR-n
- 每个 UC-n 可追溯到 US-n
- 每个 BR-n 可追溯到需求分析 BR-n
- 每个 AC-n / NAC-n 可追溯到 US-n 或非功能需求
- 每个 EX-n 可追溯到业务流程步骤

## 格式规范

- 文末「## 文档元数据」内 fenced YAML 完整（id、title、version、status、created、updated、parent、mvp_phase）；**文件开头无** `---` YAML frontmatter
- 章节结构与 prd-template.md 一致，编号连续
- Mermaid 图表语法正确，可正常渲染
- 表格格式规范，列宽合理
- 术语表（§8.1）涵盖本文档专用术语
- 变更历史（§11.2）记录初始版本
