# 设计原则与反模式

sdx-prd 技能的完整设计约束。主文件 SKILL.md 中的「核心约束」为精简版，本文件为完整规范。

---

## 设计原则

### 1. 模板驱动

输出严格遵循 [../assets/prd-template.md](../assets/prd-template.md) 的十章结构（产品概述→业务流程→产品交互→用户故事→用例模型→功能模块设计→业务规则汇总→数据字典→验收标准汇总→附录）。章节顺序与编号不可调整；无内容的章节保留标题并标注「不适用」或「待补充」。

### 2. 证据优先

用户故事、业务流程与业务规则须引用需求分析文档（FR-n、BR-n）与 `knowledge/` 事实，禁止凭空臆测。引用格式遵循项目规范：

| 证据类型 | 格式 | 示例 |
|----------|------|------|
| 功能需求 | `FR-{NNN}` | `FR-001 创建申诉单` |
| 业务规则 | `BR-{NNN}` | `BR-001 申诉时效校验` |
| 解决方案目标 | `G-{N}` | `G-1 提升申诉处理效率` |
| 知识库实体 | `{视角}-{ID}` | `BC-001 FeeAppealContext` |
| 文档章节 | `{文件} §{章节}` | `REQUIREMENT-20260327-001 §2.1` |

### 3. 按需加载

仅在本轮任务需要时打开文件：

- 步骤 1 读需求分析文档的功能需求（§2）与业务规则（§4）
- 步骤 2 按用户故事涉及的业务规则打开 `knowledge/business/`
- 步骤 3 综合步骤 1–2 产出构建用例模型
- 步骤 4 按模块涉及的交互打开 `knowledge/product/`

禁止为「完整性」通读 `knowledge/**` 或全仓源码。

### 4. 歧义标注

对需求分析文档或知识库中的歧义、矛盾、信息缺失，标注为待澄清项，在文档中显式记录。禁止自行假设缺失信息后继续，应暂停并请求用户澄清。

### 5. 范围边界

本技能仅负责输出 PRD 文档。不产出 ADD、测试设计或代码——那些是下游 sdx-design / sdx-test 的职责。

### 6. MVP 聚焦

PRD 仅覆盖目标 MVP 阶段（`--mvp` 参数指定）范围内的功能需求。不得将后续 MVP 的需求混入当前 PRD；跨 MVP 的共享前提在产品概述中简要说明即可。

### 7. 可追溯性

- 每个用户故事（US-n）可追溯到需求分析的功能需求（FR-n）
- 每个用例（UC-n）可追溯到用户故事（US-n）
- 每个业务规则（BR-n）可追溯到需求分析的业务规则定义
- 每个验收标准（AC-n / NAC-n）可追溯到用户故事或非功能需求
- 每个异常场景（EX-n）可追溯到业务流程步骤

### 8. ID 编号规范

PRD 文档使用以下 ID 编号体系：

| 前缀 | 用途 | 示例 |
|------|------|------|
| `PRD-{ID}-{N}` | 文档编号 | `PRD-20260327-001-1` |
| `US-{NNN}` | 用户故事编号 | `US-001`、`US-002` |
| `UC-{NNN}` | 用例编号 | `UC-001`、`UC-002` |
| `BR-{NNN}` | 业务规则编号（继承自需求分析） | `BR-001`、`BR-002` |
| `EX-{NNN}` | 异常场景编号 | `EX-001`、`EX-002` |
| `AC-{NNN}` | 功能验收标准编号 | `AC-001`、`AC-002` |
| `NAC-{NNN}` | 非功能验收标准编号 | `NAC-001`、`NAC-002` |

## 反模式清单（禁止）

| 反模式 | 说明 |
|--------|------|
| 臆测需求 | 未引用需求分析 FR-n 即自行补充用户故事或业务流程 |
| 吞没歧义 | 将不明确需求自行补全后继续，不标注待澄清 |
| 范围蔓延 | 在 PRD 中直接写出技术设计（接口定义、数据库表设计等） |
| 模板跳章 | 跳过或重排 prd-template.md 的章节结构 |
| 通读全库 | 一次性读取全仓知识库而不考虑本轮任务实际需要 |
| 无编号引用 | 用户故事/用例/业务规则/验收标准无统一编号，导致不可追溯 |
| MVP 越界 | 将非当前 MVP 范围的功能需求纳入 PRD |
| 故事堆砌 | 用户故事仅复制 FR-n 描述而未展开为具体场景与验收标准 |
| 空章提交 | 章节标题下无任何内容且未标注「不适用」或「待补充」 |
| 规则散落 | 业务规则散布在流程/故事/用例中未在 §7 汇总 |

## 质量标准

### 完整性

- 产品概述引用需求分析文档与 MVP 范围
- 核心业务流程与分支/异常流程完整覆盖
- 所有 MVP 范围内的 FR-n 均已转化为用户故事
- 每个用户故事有 INVEST 检查、验收标准（Given-When-Then）
- 用例模型覆盖所有参与角色与核心交互
- 功能模块划分完整，模块间关系清晰
- 业务规则汇总无遗漏，规则间无冲突
- 验收标准覆盖功能与非功能维度

### 一致性

- 用户故事与需求分析 FR-n 一一对应，无遗漏
- 业务规则与需求分析 BR-n 及 `knowledge/business/` 无矛盾
- 业务流程与需求分析中的核心场景描述一致
- 文档内部 US-n / UC-n / BR-n / EX-n / AC-n 编号连续无断号
- 各章节交叉引用自洽（故事→用例→规则→验收）

### 可行性

- 用户故事粒度合理，可在单个迭代内交付
- 业务流程可实现，无逻辑死循环或不可达状态
- 验收标准可度量、可测试
- 交互设计在现有技术框架下可实现

### 可追溯性

- US-n → FR-n（需求分析功能需求）
- UC-n → US-n（用户故事）
- BR-n → 需求分析 BR-n
- AC-n / NAC-n → US-n 或非功能需求
- EX-n → 业务流程步骤

## 最佳实践

### 业务流程设计

- 从需求分析的核心场景出发，先绘制主干流程再补充分支与异常
- 每个流程步骤标注参与角色、输入、处理逻辑、输出与关联业务规则
- 跨系统交互使用时序图（Mermaid sequenceDiagram）清晰展示调用链
- 异常场景编号与处理方式独立成表（EX-n）

### 用户故事编写

- 严格遵循 INVEST 原则：Independent / Negotiable / Valuable / Estimable / Small / Testable
- 验收标准使用 Given-When-Then 格式，每个故事至少包含正常与一个异常场景
- 故事点估算参考团队历史速率，不凭空给出数字
- 优先级继承需求分析 FR-n 的 P0–P3 分级

### 用例建模

- 用例图覆盖所有参与角色与核心交互
- 每个用例的主成功场景步骤不超过 9 步（认知负荷限制）
- 扩展场景聚焦业务异常，技术异常留给 ADD
- 用例与用户故事保持双向映射，避免孤立用例

### 交互设计

- 信息架构从用户任务目标出发，避免功能堆砌
- 操作流程标注关键校验节点与用户反馈
- 对关键页面提供交互说明或线框图链接
- 多端适配与无障碍要求显式记录

## 错误处理

| 错误场景 | 处理方式 |
|----------|----------|
| 需求分析文档不存在 | 终止，提示先执行 sdx-analysis |
| 需求分析文档中无对应 MVP 章节 | 终止，提示检查 MVP 编号或先完成 MVP 拆分 |
| 需求分析文档结构不完整 | 发出警告，列出缺失章节，基于已有内容继续并标注风险 |
| knowledge 目录缺失 | 发出警告，仅基于需求分析文档完成 PRD，标注缺少知识库基线 |
| 模板文件不存在 | 终止，提示确认 `.ai/skills/sdx-prd/assets/prd-template.md` 存在 |
| FR-n 与 US-n 映射不完整 | 暂停，输出未覆盖的 FR-n 清单，请求用户确认是否遗漏 |
| 输出目录不存在 | 自动创建 `system/requirements/REQUIREMENT-{ID}/MVP-{N}/` 目录 |
