# 需求分析阶段规范

## 1 阶段目标

基于解决方案文档和系统说明文档，进行深度研究与探索，将高层解决方案细化为具体的、可执行的需求分析，合理拆分为多个MVP阶段，确保每个MVP具备独立交付价值。

### 2 输入与输出

**输入**：

- 解决方案文档（`solutions/`）
- 知识库（`knowledge/`）
- 规约文件（`specs/`）

**输出**：

- 需求分析文档（`analysis/REQUIREMENT-{ID}.md`）

#### 步骤 1：深度研究与探索

**Agent**：requirements-analyst
> 自动匹配Agent符合的角色和SubAgent，选用相应的Skills

**任务描述**：

```markdown
## 任务：基于解决方案进行深度研究与探索

### 输入
- 解决方案文档
- 系统说明文档
- 相关服务规约和领域规约

### 工作要求
1. **业务领域研究**：
   - 梳理涉及的业务领域及其边界
   - 分析领域内的核心业务概念和规则
   - 识别跨领域的交互点和数据流转

2. **现有实现探索**：
   - 研究现有系统中相关功能的实现方式
   - 识别可复用的组件、模式和基础设施
   - 发现现有实现中的技术债务和改进空间

3. **行业实践参考**：
   - 参考行业最佳实践和设计模式
   - 评估适用于当前场景的技术方案

4. **边界场景探索**：
   - 识别边界条件和异常场景
   - 分析高并发、大数据量等极端场景的处理需求
   - 考虑数据一致性、幂等性等分布式系统关键问题

### 输出格式
深度研究报告（作为需求分析文档的输入）
```

#### 步骤 2：需求细化与建模

**Agent**：requirements-analyst
> 自动匹配Agent符合的角色和SubAgent，选用相应的Skills

**任务描述**：

```markdown
## 任务：将解决方案细化为具体需求

### 输入
- 解决方案文档
- 步骤1的深度研究报告

### 工作要求
1. **功能需求细化**：
   - 将每个业务场景分解为具体的功能需求
   - 为每个功能需求定义清晰的输入、处理逻辑、输出
   - 标注功能需求的优先级（P0/P1/P2/P3）

2. **非功能需求明确**：
   - 性能需求（响应时间、吞吐量、并发量）
   - 可用性需求（SLA目标、容灾策略）
   - 安全需求（认证、授权、数据保护）
   - 可观测性需求（日志、监控、告警）
   - 兼容性需求（向前/向后兼容）

3. **业务规则提取**：
   - 提取所有涉及的业务规则
   - 用结构化方式描述每条规则的触发条件和执行逻辑
   - 标注规则的优先级和冲突处理策略

4. **数据需求分析**：
   - 识别新增/变更的数据实体
   - 分析数据的生命周期（创建、更新、归档、删除）
   - 明确数据一致性要求

### 输出格式
细化需求清单（作为需求分析文档的章节）
```

#### 步骤 3：MVP 拆分与规划

**Agent**：requirements-analyst

**任务描述**：

```markdown
## 任务：将需求合理拆分为多个MVP阶段

### 输入
- 步骤2的细化需求清单
- 解决方案中的MVP拆分建议

### 工作要求
1. **MVP拆分原则**：
   - 每个MVP必须具备独立的业务价值
   - 每个MVP必须可独立部署和验证
   - MVP之间的依赖关系清晰且单向
   - 优先交付核心业务路径，渐进增强

2. **MVP内容规划**：
   - 为每个MVP定义明确的范围和交付目标
   - 列出每个MVP包含的功能需求清单
   - 定义每个MVP的验收标准
   - 估算每个MVP的工作量和交付周期

3. **MVP排序策略**：
   - 基于业务价值、技术依赖和风险的综合排序
   - 高风险/高不确定性的内容优先验证
   - 基础能力优先于上层业务功能

### 输出格式
MVP拆分方案（作为需求分析文档的核心章节）
```

#### 步骤 4：依赖分析与风险评估

**Agent**：requirements-analyst 和 （quality-guardian 或 quality-engineer）

**任务描述**：

```markdown
## 任务：分析MVP间的依赖关系并评估风险

### 工作要求
1. **依赖关系分析**：
   - MVP之间的功能依赖
   - MVP之间的数据依赖
   - MVP之间的接口依赖
   - 外部系统/团队依赖

2. **风险评估**：
   - 技术风险（新技术引入、复杂集成等）
   - 业务风险（需求理解偏差、业务规则变更等）
   - 进度风险（依赖延迟、资源不足等）
   - 质量风险（测试覆盖不足、回归影响等）

3. **风险应对策略**：
   - 为每个高风险项制定应对方案
   - 定义风险监控指标和预警机制
```
