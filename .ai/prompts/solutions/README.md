# 解决方案阶段规范

本文档定义了 AI SDD 解决方案阶段的目标、输入输出、Agent 工作流及文档模板。

## 1. 阶段目标

从海量、模糊甚至矛盾的业务描述中，提取结构化需求，结合现存系统状态，评估业务影响面，识别潜在冲突，确立业务目标和解决思路，输出高质量的解决方案文档。

## 2. 输入与输出

**输入**：

- 业务需求描述（可能是邮件、会议纪要、口头描述、工单等非结构化信息）
- 现存系统说明文档（`docs/instructions/`）
- 现存规约文件（`specs/`）
- 历史解决方案文档（`docs/solutions/archive/`，可选参考）

**输出**：

- 解决方案文档（`docs/solutions/SOLUTION-{ID}.md`）

### 步骤 1：需求提取与结构化

**Agent**：solution-analyst 或 requirements-analyst
> 自动匹配Agent符合的角色和SubAgent，选用相应的Skills

**任务描述**：

```markdown
## 任务：从业务描述中提取结构化需求

### 输入
- 原始业务描述/需求来源

### 工作要求
1. **信息萃取**：从非结构化的业务描述中，识别并提取以下要素：
   - 业务背景与动机（为什么要做）
   - 业务目标与期望价值（做了有什么好处）
   - 核心业务场景（要做什么）
   - 涉及的用户角色（谁来用）
   - 关键约束条件（有什么限制）
   - 时间要求与优先级（什么时候要）

2. **歧义标注**：对于模糊、矛盾或缺失的信息，明确标注并提出澄清问题

3. **需求初步分类**：
   - 功能性需求 vs 非功能性需求
   - 新增功能 vs 变更功能 vs 修复问题
   - 业务需求 vs 技术需求

### 输出格式
结构化需求提取报告（作为解决方案文档的输入）
```

### 步骤 2：影响面评估与分析

**Agent**：(solution-analyst 或 requirements-analyst) 和 (quality-guardian 或 quality-engineer)
> 自动匹配Agent符合的角色和SubAgent，选用相应的Skills

**任务描述**：

```markdown
## 任务：评估需求对现存业务的影响面
- 生成受影响的功能列表及影响程度（高/中/低）
- 标注影响传播路径（直接影响 → 间接影响）

### 输出格式
影响面评估报告（作为解决方案文档的章节）
```

### 步骤 3：冲突识别与化解

**Agent**：solution-analyst 或 requirements-analyst
> 自动匹配Agent符合的角色和SubAgent，选用相应的Skills

**任务描述**：

```markdown
## 任务：识别潜在的业务冲突与系统冲突

### 输入
- 步骤1的结构化需求
- 步骤2的影响面评估
- 现有业务规则文档
- 进行中的其他需求/解决方案（如有）

### 工作要求
1. **业务冲突识别**：
   - 新需求与现有业务规则的冲突
   - 新需求与正在进行的其他需求的冲突
   - 业务流程变更导致的上下游影响

2. **系统冲突识别**：
   - 模型冲突（同一实体的不同变更方向）
   - 契约冲突（不兼容的变更）
   - 资源竞争冲突（共享基础设施的变更）

3. **冲突化解建议**：
   - 对每个识别的冲突，提出化解方案
   - 评估化解方案的成本 and 风险

### 输出格式
冲突分析报告（作为解决方案文档的章节）
```

### 步骤 4：方案制定与评估

**Agent**：solution-analyst 或 requirements-analyst
> 自动匹配Agent符合的角色和SubAgent，选用相应的Skills

**任务描述**：

```markdown
## 任务：制定解决方案并评估可行性

### 输入
- 步骤1-3的所有分析结果

### 工作要求
1. **明确业务目标**：
   - 量化的业务目标（可度量的成功标准）
   - 业务价值评估

2. **解决思路阐述**：
   - 整体解决思路与策略
   - 关键决策点及推荐方向
   - 备选方案对比（如有多种方案）

3. **可行性评估**：
   - 技术可行性（系统架构、技术栈、组件）
   - 资源可行性（时间、人力、基础设施）
   - 风险评估与应对策略

4. **初步范围界定**：
   - 本次解决方案的范围边界（做什么、不做什么）
   - 初步的MVP-Phase拆分建议

### 输出格式
解决方案核心内容（作为解决方案文档的主体）
```

**质量门禁检查项**：

```yaml
solution_quality_gate:
  completeness:
    - "业务背景与动机是否清晰"
    - "业务目标是否可度量"
    - "影响面评估是否覆盖所有受影响服务"
    - "冲突分析是否完整"
    - "解决思路是否明确"
    - "范围边界是否清晰"
  consistency:
    - "需求描述与现有系统文档无矛盾"
    - "影响面评估与系统架构文档一致"
  feasibility:
    - "技术方案在现有架构下可实现"
    - "资源估算合理"
  traceability:
    - "每个业务目标可追溯到原始需求"
    - "每个影响点可追溯到具体服务/模块"
```
