# 第六部分：元数据标签与规范体系

## 6.1 元数据标签体系总体设计

### 6.1.1 设计目标

元数据标签体系是知识文档库的"神经系统"，承担三大核心职能：

1. **可发现性（Discoverability）**：支持从任意维度快速定位相关知识片段
2. **可关联性（Traceability）**：建立四视角之间的显式追溯链路
3. **可推导性（Derivability）**：为未来的需求分析和方案设计提供机器可推理的结构化元信息

### 6.1.2 标签分类框架

```
元数据标签体系
├── 结构性标签（Structural Tags）      ── 描述知识实体在体系中的位置
├── 语义性标签（Semantic Tags）         ── 描述知识实体的业务与技术含义
├── 关联性标签（Relational Tags）       ── 描述知识实体之间的映射与依赖
├── 治理性标签（Governance Tags）       ── 描述知识实体的生命周期与质量状态
├── 演进性标签（Evolution Tags）        ── 描述知识实体的变更历史与未来规划
└── 场景性标签（Scenario Tags）         ── 为需求分析/方案设计等消费场景预留
```

## 6.2 各类标签详细定义

### 6.2.1 结构性标签（Structural Tags）

结构性标签用于定位知识实体在四视角层级体系中的精确坐标。

```yaml
# 结构性标签 Schema 定义
structural_tags:
  # ── 业务视角坐标 ──
  business_coordinates:
    domain_id: string              # 业务域标识，如 "BD-ORDER"
    domain_name: string            # 业务域名称
    subdomain_id: string           # 子域标识，如 "BSD-ORDER-FULFILLMENT"
    subdomain_name: string
    bounded_context_id: string     # 限界上下文标识，如 "BC-ORDER-MGMT"
    bounded_context_name: string
    aggregate_id: string           # 聚合标识（如适用）
    strategic_classification: enum # core_domain / supporting_domain / generic_domain

  # ── 产品视角坐标 ──
  product_coordinates:
    product_line_id: string        # 产品线标识，如 "PL-ECOMMERCE"
    product_line_name: string
    module_id: string              # 产品模块标识，如 "PM-SHOPPING-CART"
    module_name: string
    feature_id: string             # 功能点标识（如适用）
    module_type: enum              # frontend / middle_platform / backend / cross_cutting

  # ── 技术视角坐标 ──
  technical_coordinates:
    system_id: string              # 系统标识，如 "SYS-ECOMMERCE"
    system_name: string
    subsystem_id: string           # 子系统标识，如 "SUBSYS-ORDER"
    subsystem_name: string
    application_id: string         # 应用标识，如 "APP-ORDER-SERVICE"
    application_name: string
    repo_url: string               # 代码仓库地址
    microservice_id: string        # 微服务标识（如适用）

  # ── 数据视角坐标 ──
  data_coordinates:
    data_store_id: string          # 数据存储标识，如 "DS-ORDER-DB"
    data_entity_id: string         # 数据实体标识（如适用）
    data_classification: enum      # master_data / transaction_data / reference_data / analytical_data
```

**使用示例**：

```yaml
# 一个微服务文档的结构性标签
---
id: "MS-ORDER-COMMAND"
name: "订单命令服务"
structural_tags:
  business_coordinates:
    domain_id: "BD-ORDER"
    domain_name: "订单域"
    subdomain_id: "BSD-ORDER-FULFILLMENT"
    subdomain_name: "订单履约"
    bounded_context_id: "BC-ORDER-MGMT"
    bounded_context_name: "订单管理"
    strategic_classification: core_domain
  product_coordinates:
    product_line_id: "PL-ECOMMERCE"
    product_line_name: "电商平台"
    module_id: "PM-ORDER-MGMT"
    module_name: "订单管理模块"
    module_type: backend
  technical_coordinates:
    system_id: "SYS-ECOMMERCE"
    subsystem_id: "SUBSYS-ORDER"
    application_id: "APP-ORDER-SERVICE"
    repo_url: "https://git.example.com/order-service"
    microservice_id: "MS-ORDER-COMMAND"
  data_coordinates:
    data_store_id: "DS-ORDER-DB"
    data_classification: transaction_data
---
```

### 6.2.2 语义性标签（Semantic Tags）

语义性标签用于描述知识实体的业务含义、技术特征和非功能属性，支持按语义维度进行检索和聚合分析。

```yaml
semantic_tags:
  # ── 业务语义 ──
  business_semantics:
    business_capability: string       # 业务能力，如 "订单创建"、"库存扣减"
    business_value_stream: string     # 价值流，如 "客户获取"、"订单履约"
    business_scenario: [string]       # 业务场景列表，如 ["B2C下单", "B2B批量采购"]
    industry_domain: string           # 行业领域，如 "电商"、"金融"、"物流"
    regulatory_relevance: [string]    # 关联法规，如 ["GDPR", "电商法第XX条"]
    business_criticality: enum        # critical / high / medium / low

  # ── 技术语义 ──
  technical_semantics:
    architectural_style: [enum]       # microservice / event_driven / cqrs / saga / bff
    communication_pattern: [enum]     # sync_request_reply / async_event / async_command / batch
    data_consistency_model: enum      # strong / eventual / saga_based
    scalability_pattern: [enum]       # horizontal / vertical / sharding / caching
    resilience_pattern: [enum]        # circuit_breaker / retry / bulkhead / fallback / rate_limit
    security_classification: enum     # public / internal / confidential / restricted
    performance_tier: enum            # realtime(<100ms) / near_realtime(<1s) / standard(<5s) / batch

  # ── 非功能属性 ──
  quality_attributes:
    availability_target: string       # 如 "99.99%"
    rpo: string                       # Recovery Point Objective，如 "0"
    rto: string                       # Recovery Time Objective，如 "5min"
    peak_qps: string                  # 峰值 QPS
    data_volume: string               # 数据量级，如 "亿级"
    compliance_requirements: [string] # 合规要求
```

### 6.2.3 关联性标签（Relational Tags）

关联性标签是四视角融合的核心，定义知识实体之间的显式映射关系。

```yaml
relational_tags:
  # ── 显式映射声明 ──
  cross_view_mappings:
    # 业务 → 产品 映射
    realizes_features: [string]       # 当前实体实现了哪些功能点 Feature IDs
    supports_use_cases: [string]      # 当前实体支撑了哪些用例 UseCase IDs
    participates_in_processes: [string] # 参与了哪些业务流程 Process IDs
    governed_by_rules: [string]       # 受哪些业务规则约束 Rule IDs

    # 业务 → 技术 映射
    implemented_by_services: [string] # 由哪些微服务实现 Microservice IDs
    deployed_on_applications: [string] # 部署在哪些应用上 Application IDs
    exposed_via_apis: [string]        # 通过哪些 API 暴露 API IDs

    # 业务 → 数据 映射
    persisted_as_entities: [string]   # 持久化为哪些数据实体 DataEntity IDs
    stored_in: [string]               # 存储在哪些数据存储中 DataStore IDs

    # 技术 → 数据 映射
    owns_data_entities: [string]      # 拥有哪些数据实体（数据所有权）
    reads_from: [string]              # 从哪些数据源读取
    writes_to: [string]               # 向哪些数据源写入

  # ── 依赖关系声明 ──
  dependencies:
    upstream: # 上游依赖
      - target_id: string
        target_type: enum             # service / api / event / data_entity
        dependency_type: enum         # runtime / build_time / data
        coupling_level: enum          # tight / loose / eventual
        contract_ref: string          # 契约文件引用
        is_critical: boolean          # 是否关键依赖
    downstream: # 下游消费者
      - target_id: string
        target_type: enum
        dependency_type: enum
        coupling_level: enum
        contract_ref: string

  # ── 事件关联 ──
  event_associations:
    publishes_events: [string]        # 发布的领域事件 DomainEvent IDs
    subscribes_events: [string]       # 订阅的领域事件 DomainEvent IDs
    triggers_sagas: [string]          # 触发的 Saga 流程 IDs
```

**映射关系矩阵总览**：

```
                    ┌──────────┐
                    │ 业务视角  │
                    │ Business │
                    └─┬──┬──┬─┘
           realizes   │  │  │  persisted_as
          ┌───────────┘  │  └───────────┐
          ▼              │              ▼
    ┌──────────┐  implemented_by  ┌──────────┐
    │ 产品视角  │◄───────┼───────►│ 数据视角  │
    │ Product  │         │        │  Data    │
    └─┬────────┘         │        └────────┬─┘
      │                  │                 │
      │ supported_by     │    owns/reads   │
      └──────────┐       │       ┌─────────┘
                 ▼       ▼       ▼
               ┌──────────────────┐
               │    技术视角       │
               │   Technical      │
               └──────────────────┘

映射关系清单:
  业务 ↔ 产品:  BoundedContext → Feature (realizes)
                Aggregate → UseCase (supports)
                DomainEvent → BusinessProcess (triggers)
                BusinessRule → Feature (governs)

  业务 ↔ 技术:  BoundedContext → Microservice (implemented_by)
                Aggregate → Microservice (encapsulated_in)
                DomainEvent → APIEndpoint/MessageTopic (exposed_via)

  业务 ↔ 数据:  AggregateRoot → DataEntity (persisted_as)
                DomainEntity → DataEntity (persisted_as)
                ValueObject → DataField/EmbeddedDocument (embedded_in)

  产品 ↔ 技术:  Feature → Microservice (supported_by)
                UseCase → APIEndpoint (invokes)
                BusinessProcess → [Microservice] (orchestrated_by)

  产品 ↔ 数据:  Feature → DataEntity (reads/writes)
                BusinessRule → DataField (constrains)
                BusinessProcess → DataFlow (generates)

  技术 ↔ 数据:  Microservice → DataStore (owns/reads/writes)
                Application → DataStore (deployed_with)
                APIEndpoint → DataEntity (exposes/mutates)
```

### 6.2.4 治理性标签（Governance Tags）

治理性标签用于管理知识实体的生命周期、质量状态和审核流程。

```yaml
governance_tags:
  # ── 生命周期管理 ──
  lifecycle:
    status: enum                      # draft / in_review / approved / published / deprecated / archived
    maturity_level: enum              # L0_placeholder / L1_skeleton / L2_documented / L3_verified / L4_maintained
    effective_date: date              # 生效日期
    review_date: date                 # 下次审核日期
    expiry_date: date                 # 过期日期（可选）
    retirement_plan: string           # 退役计划（如适用）

  # ── 质量与完整性 ──
  quality:
    completeness_score: float         # 0.0 - 1.0，文档完整度
    completeness_gaps: [string]       # 缺失的必填部分列表
    accuracy_confidence: enum         # verified / peer_reviewed / self_assessed / unverified / outdated
    last_verified_date: date          # 最后一次验证日期
    last_verified_by: string          # 验证人
    automated_check_result: object    # 自动化检查结果
      schema_valid: boolean
      links_valid: boolean
      cross_ref_consistent: boolean

  # ── 责任归属 ──
  ownership:
    business_owner: string            # 业务负责人
    technical_owner: string           # 技术负责人
    document_maintainer: string       # 文档维护人
    review_committee: [string]        # 评审委员会成员
    team: string                      # 所属团队

  # ── 审核追踪 ──
  audit_trail:
    created_at: datetime
    created_by: string
    last_updated_at: datetime
    last_updated_by: string
    change_history:                   # 变更历史摘要
      - version: string
        date: datetime
        author: string
        summary: string
        change_type: enum             # major / minor / patch / metadata_only
```

### 6.2.5 演进性标签（Evolution Tags）

演进性标签记录知识实体的变更轨迹和未来规划，是支撑系统改造和影响分析的关键。

```yaml
evolution_tags:
  # ── 变更追溯 ──
  change_tracking:
    current_version: string           # 当前版本号（语义化版本）
    previous_versions: [string]       # 历史版本列表
    breaking_changes_since: string    # 最近一次破坏性变更的版本
    changelog_ref: string             # 变更日志文件引用
    related_adr: [string]             # 关联的架构决策记录 ADR IDs
    migration_guide_ref: string       # 迁移指南引用（如适用）

  # ── 未来规划 ──
  roadmap:
    planned_changes: [PlannedChange]  # 计划中的变更
    target_state_ref: string          # 目标态文档引用
    tech_debt_items: [TechDebtItem]   # 技术债务项
    improvement_opportunities: [string] # 改进机会

  # ── 兼容性声明 ──
  compatibility:
    backward_compatible_since: string # 向后兼容起始版本
    api_versioning_strategy: enum     # url_path / header / query_param / content_negotiation
    deprecation_notice:               # 废弃通知
      deprecated_since: string
      sunset_date: date
      replacement_ref: string         # 替代方案引用
      migration_effort: enum          # trivial / minor / moderate / major / rewrite

# 子结构定义
PlannedChange:
  id: string
  summary: string
  target_version: string
  planned_date: date
  impact_scope: [string]              # 受影响的实体 IDs
  change_type: enum                   # enhancement / refactoring / migration / deprecation
  priority: enum                      # P0 / P1 / P2 / P3
  requirement_ref: string             # 关联需求编号

TechDebtItem:
  id: string
  description: string
  severity: enum                      # critical / high / medium / low
  estimated_effort: string            # 预估工作量
  impact_if_unresolved: string        # 不解决的影响
  proposed_solution: string           # 建议解决方案
```

### 6.2.6 场景性标签（Scenario Tags）

场景性标签是为未来"需求分析推导"和"解决方案生成"预留的核心标签集，使知识库具备支撑自动化推理的能力。

```yaml
scenario_tags:
  # ── 需求分析支撑 ──
  requirement_analysis_support:
    affected_by_requirement_types: [enum]  # 该实体可能被哪类需求影响
      # new_feature / enhancement / bug_fix / performance / security
      # compliance / data_migration / integration / deprecation
    change_sensitivity: enum               # 变更敏感度
      # high_frequency / medium_frequency / low_frequency / stable
    impact_radius: enum                    # 变更影响半径
      # isolated / module_scope / subsystem_scope / system_scope / cross_system
    typical_change_patterns: [string]      # 典型变更模式描述
      # 如 "新增支付方式" → 影响 PaymentGateway + OrderService + ...
    prerequisite_knowledge: [string]       # 理解该实体需要的前置知识 Entity IDs

  # ── 解决方案设计支撑 ──
  solution_design_support:
    reusable_patterns: [string]            # 可复用的设计模式引用
    known_constraints: [string]            # 已知约束条件
    integration_points: [IntegrationPoint] # 集成点清单
    capacity_baseline: object              # 容量基线数据
      current_qps: number
      current_data_volume: string
      growth_rate: string
    risk_factors: [RiskFactor]             # 风险因素

  # ── 影响分析支撑 ──
  impact_analysis_support:
    blast_radius_map:                      # 爆炸半径映射
      direct_dependencies: [string]        # 直接依赖方 IDs
      transitive_dependencies: [string]    # 传递依赖方 IDs
      data_dependencies: [string]          # 数据依赖方 IDs
      process_dependencies: [string]       # 流程依赖方 IDs
    rollback_complexity: enum              # trivial / moderate / complex / infeasible
    testing_strategy_hint: [enum]          # unit / integration / contract / e2e / performance / chaos

# 子结构定义
IntegrationPoint:
  target_id: string
  integration_type: enum                   # api_call / event / shared_db / file_transfer / etl
  protocol: string
  contract_ref: string
  sla: object
  failure_mode: string                     # 失败模式描述
  fallback_strategy: string               # 降级策略

RiskFactor:
  description: string
  probability: enum                        # high / medium / low
  impact: enum                             # critical / major / moderate / minor
  mitigation: string                       # 缓解措施
```

## 6.4 元数据标签的分级填写要求

并非所有标签在所有阶段都必须完整填写。定义三级填写要求：

| 级别 | 适用阶段 | 必填标签范围 | 完整度目标 |
|------|----------|-------------|-----------|
| **L1 - 最小可用** | 知识库初建 / 新增文档 | 基础标识 + 结构性标签（至少一个视角坐标）+ 治理性标签（status, owner） | ≥ 30% |
| **L2 - 关联可查** | 知识库完善期 | L1 + 关联性标签（核心映射）+ 语义性标签（business_criticality, security） | ≥ 60% |
| **L3 - 推导可用** | 支撑需求分析/方案设计 | L2 + 场景性标签 + 演进性标签 + 完整的交叉映射 | ≥ 85% |

## 6.5 标签一致性校验规则

```yaml
# 一致性校验规则定义
validation_rules:
  # 规则1：结构坐标完整性
  - id: "VAL-001"
    name: "至少一个视角坐标完整"
    severity: error
    rule: |
      structural_tags 中至少有一个视角的坐标填写完整
      （即 business_coordinates 或 technical_coordinates 中的
       必填字段不为空）

  # 规则2：双向映射一致性
  - id: "VAL-002"
    name: "交叉映射双向一致"
    severity: warning
    rule: |
      如果 A 的 relational_tags.cross_view_mappings.implemented_by_services
      包含 B，则 B 的 relational_tags.cross_view_mappings 中必须存在
      对 A 的反向引用

  # 规则3：生命周期状态合理性
  - id: "VAL-003"
    name: "已发布文档必须有 owner"
    severity: error
    rule: |
      当 governance_tags.lifecycle.status == "published" 时，
      ownership.technical_owner 和 ownership.document_maintainer
      不能为空

  # 规则4：废弃实体必须有替代引用
  - id: "VAL-004"
    name: "废弃实体需声明替代方案"
    severity: warning
    rule: |
      当 governance_tags.lifecycle.status == "deprecated" 时，
      evolution_tags.compatibility.deprecation_notice.replacement_ref
      不能为空

  # 规则5：关键依赖必须有契约引用
  - id: "VAL-005"
    name: "关键上游依赖需有契约"
    severity: warning
    rule: |
      当 dependencies.upstream[].is_critical == true 时，
      contract_ref 不能为空

  # 规则6：跨视角坐标一致性
  - id: "VAL-006"
    name: "技术坐标与业务坐标映射一致"
    severity: info
    rule: |
      如果 microservice 声明了 bounded_context_id，
      则该 bounded_context 的文档中应包含对该 microservice 的引用
```

## 6.8 标签命名规范

### 6.8.1 ID 命名规则

```
规则：{类型前缀}-{业务域缩写}-{具体名称}
格式：全大写，单词间用连字符分隔

类型前缀对照表：
  BD    → Business Domain（业务域）
  BSD   → Business SubDomain（业务子域）
  BC    → Bounded Context（限界上下文）
  AGG   → Aggregate（聚合）
  PL    → Product Line（产品线）
  PM    → Product Module（产品模块）
  FT    → Feature（功能点）
  UC    → Use Case（用例）
  BP    → Business Process（业务流程）
  BR    → Business Rule（业务规则）
  SYS   → System（系统）
  SUBSYS→ Subsystem（子系统）
  APP   → Application（应用）
  MS    → Microservice（微服务）
  API   → API Endpoint（API端点）
  DE    → Data Entity（数据实体）
  DS    → Data Store（数据存储）
  DF    → Data Flow（数据流）
  EVT   → Domain Event（领域事件）
  ADR   → Architecture Decision Record（架构决策记录）

示例：
  BD-ORDER              → 订单业务域
  BC-ORDER-MGMT         → 订单管理限界上下文
  AGG-ORDER             → 订单聚合
  MS-ORDER-COMMAND      → 订单命令微服务
  API-ORDER-CREATE      → 创建订单API
  DE-ORDER              → 订单数据实体
  EVT-ORDER-CREATED     → 订单已创建事件
  ADR-ORDER-001         → 订单域第001号架构决策
```

### 6.8.2 版本号规范

```
采用语义化版本号（Semantic Versioning）：MAJOR.MINOR.PATCH

MAJOR：不兼容的重大变更（如领域模型重构、API破坏性变更）
MINOR：向后兼容的功能新增（如新增字段、新增API端点）
PATCH：向后兼容的修正（如文档纠错、描述优化）

示例：
  0.1.0 → 初始草稿
  0.2.0 → 补充了关联标签
  1.0.0 → 首次正式发布
  1.1.0 → 新增了一个API端点描述
  1.1.1 → 修正了数据模型中的字段描述
  2.0.0 → 订单聚合模型重构
```
